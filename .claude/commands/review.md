# Ревью PR

Прогон ревью для: $ARGUMENTS

`$ARGUMENTS` — номер PR (`19`) или пусто (тогда ревью текущей ветки против `origin/main`).

## Назначение

Оркестратор спавнит 3-4 параллельных read-only субагента с разными оптиками, собирает findings, дедупает, выводит структурированный отчёт. Цель — довести PR до состояния «грамотно реализовано», а не плодить tech-debt.

## CRITICAL RULES (NO EXCEPTIONS)

- **Subagent isolation.** Каждый субагент стартует холодным через `Agent` tool. Inline-ревью запрещено.
- **Параллельный запуск.** Все применимые субагенты — в **ОДНОМ message** с несколькими `Agent` tool_use блоками. Sequential — только Шаги 7→8.
- **DEFAULT = fix in this PR.** Любое finding по умолчанию = `fix-now`. Punt в issue — только по 5 критериям ниже.
- **Self-check defer'ов перед HALT'ом.** Каждый предложенный `defer-to-issue` оркестратор валидирует против критериев. Не прошёл (severity `minor`/`nitpick`) — переклассифицируй в `fix-now` молча. Не прошёл (`blocker`/`major`) — оставь в отчёте с пометкой `[требует одобрения]`.
- **VETO Алихана** — выше всех правил. Его слово «fix-now» → fix-now независимо от severity.
- **READ-ONLY на код.** Edit/Write/NotebookEdit на исходники запрещены. Можно: Read, Bash, gh. После approval Алихана разрешено: (a) обновлять `docs/standards/<стандарт>.md` (правила) и/или `docs/standards/review-checklist.md` (только при новом слое / Hard rule / стандарт-файле); (b) постить inline-комменты в PR через `gh api`.
- **АВТОНОМНОСТЬ.** Шаги 1-7 выполняются non-stop, без промежуточных подтверждений. Единственная HALT-точка — Шаг 8.

## Когда `defer-to-issue` ДОПУСТИМ

5 закрытых критериев:

1. **Architectural refactor** — меняет публичный контракт между слоями (handler↔service↔repository) или cross-cutting concern (auth, logging, transactions, codegen pipeline, RepoFactory shape) **И** один из backstop'ов: либо >15 файлов, либо >800 строк net diff. Правка 5–15 файлов БЕЗ изменения межслойного контракта — `fix-now`.
2. **Data migration** — требует миграции существующих production данных (не схемы — данных).
3. **External dependency** — блокировано внешним сервисом / контрактом / другим репо.
4. **Out-of-PR-scope** — finding про код, который не затронут diff'ом и не вытекает напрямую из изменений PR. **Исключение:** `regression`-finding от acceptance-auditor — всегда `fix-now`, даже если сломанный код вне diff'а.
5. **Operational task** — задача без кодового изменения (конвертация ассетов, ротация секрета).

При сомнении — всегда `fix-now`. Список закрытый. Расширение — через явное согласование с Алиханом + правка этой секции.

`blocker`/`major`-defer'ы выводятся в отчёт с пометкой `[требует одобрения]`. Без явного действия по умолчанию = `fix-now`.

## Шкала severity

Severity finding'а определяется `[severity]`-меткой буллета `## Что ревьюить` соответствующего стандарта. Subagent может escalate / downgrade с обоснованием в `rationale`.

- **blocker** — security/PII, regression, mutate без аудита, утечка секрета, `[blocker]`-буллет стандарта.
- **major** — `[major]`-буллет стандарта, мёртвый код, дублирование, говнокод, недотест критичной ветки.
- **minor** — `[minor]`-буллет стандарта, стилистика, точечная неоптимальность, missing edge-case test.
- **nitpick** — субъективная стилистика, naming, комментарии, `[nitpick]`-буллет стандарта.

## Маркировка агентских комментов

Аккаунт GitHub у Алихана и review-agent'а **общий** — login-фильтр не работает. Единственный способ отличить агентские комменты — маркер.

**Каждый коммент**, постимый review-agent'ом в PR (inline-коммент, общий PR-коммент, ответ в треде), **обязан начинаться с префикса `[🤖 review-agent]`** в первой строке body. Без исключений.

Распознавание: regex `^\[🤖 review-agent\]` по `.body` коммента.

- `body` начинается с маркера → коммент агента (наш собственный, прошлый прогон).
- `body` НЕ начинается с маркера → коммент Алихана (живой человек).

Issues от агента маркером **не помечаются** — там путаницы нет, issue либо есть, либо нет.

## Кандидаты в стандарты — СТРОГИЙ ФИЛЬТР

Стандарты не раздуваем. Default = **отбросить**. Кандидат принимается ТОЛЬКО если выполнены ВСЕ 5 критериев:

1. **Паттерн, не разовый случай.** В текущем PR ≥3 примера, ИЛИ субагент через `grep` нашёл ≥3 примера в существующем коде.
2. **Severity ≥ `major`.** `minor`/`nitpick`-кандидаты отбрасываются молча.
3. **Конкретность и проверяемость.** Правило можно проверить grep'ом / линтером / по `path:line`. «Пишите хороший код» — не правило.
4. **Одно предложение, без оговорок.** Если для формулировки нужны условия «кроме случая X, кроме Y» — слишком сырое, отбросить.
5. **Не дублирует существующее.** Перед предложением Алихану оркестратор проверяет `docs/standards/` через `grep` — есть ли уже похожее правило.

Оркестратор валидирует каждого кандидата против всех 5 критериев на Шаге 6. Не прошёл хоть один — отбрасывается **молча, без вопроса Алихану**. В Шаг 7 идут ТОЛЬКО кандидаты, прошедшие фильтр.

## Шаги выполнения

### Шаг 1 — Сбор контекста

#### 1.1 Зафиксируй repo + identity

```bash
git fetch origin
REPO=$(gh repo view --json nameWithOwner --jq .nameWithOwner)
```

`REPO` подставляется во все `gh api`-вызовы. Литерал `{owner}/{repo}` в `gh api` НЕ работает.

#### 1.2 Определи режим и команду для diff'а

- **PR номер передан** → режим `pr`, команда для агентов: `gh pr diff <N>`. Дополнительно: `git fetch origin "+refs/pull/<N>/head:refs/remotes/origin/pr-<N>"` (чтобы агенты могли при нужде делать `git show pr-<N>:<path>` для полного файла).
- **PR номера нет** → режим `branch`, команда для агентов: `git diff $(git merge-base origin/main HEAD)..HEAD`.

Проверь, что diff не пустой — выполни команду сам, если empty — сообщи и завершись.

#### 1.3 Существующие inline-комменты PR (только режим `pr`)

```bash
gh api repos/$REPO/pulls/<N>/comments --paginate \
  --jq '.[] | {id, path, line: (.line // .original_line), user: .user.login, body, in_reply_to_id, created_at, html_url}'
```

Сохрани в памяти как `EXISTING_COMMENTS`. Используется в трёх местах:
- **Шаг 2** — передаётся в промпт каждому субагенту как «уже выявлено, не дублируй».
- **Шаг 3** — финальный дедуп findings с уже существующими комментами.
- **Шаг 1.4** — для определения «живых тредов».

#### 1.4 Живые vs закрытые треды (только режим `pr`)

Группируй `EXISTING_COMMENTS` по `in_reply_to_id` в треды.

- **«Коммент агента»** = `body` начинается с `[🤖 review-agent]` (regex: `^\[🤖 review-agent\]`).
- **«Коммент Алихана»** = всё остальное (живой человек; см. секцию «Маркировка агентских комментов»).
- **Закрытый тред** = последний коммент Алихана начинается с approval-маркера. Regex (case-insensitive): `^\s*(ок|согласен|норм|пойдёт|гуд|принято|✅)([\s.,!]|$)`.
- **Живой тред** = последний коммент Алихана НЕ approval (вопрос, возражение).
- **В треде только комменты агента** (Алихан ни разу не отвечал) → живой с пометкой `(ожидает реакции Алихана)`.
- **Последний коммент агента, но был прежний approval Алихана** → закрытый.
- Из закрытых тредов извлеки `[чеклист-кандидат]` маркеры (regex `\[чеклист-кандидат\]`).

#### 1.5 Метаданные PR + спека

- Режим `pr`: `gh pr view <N> --json number,title,body,baseRefName,headRefName,url`.
- Спека/PRD: проверь `_bmad-output/implementation-artifacts/spec-*.md`, `docs/prd*.md`. Plan-файлы спеками НЕ считаются.
- **Матчинг спеки с PR.** Спека релевантна, если: упомянута в PR body, тематически совпадает с branch name, или явная ссылка из плана. Без матчинга — НЕ передавай спеку acceptance-auditor / edge-case-hunter.
- Низкая уверенность матчинга — пометь в отчёте под «Сводка», субагентам не передавай.

### Шаг 2 — Параллельный multi-subagent ревью

В **ОДНОМ message** запусти все применимые субагенты через `Agent` tool. **Каждому передай полный шаблон ниже**, подставив переменные.

Минимальный набор — всегда:
- `standards-auditor`
- `blind-hunter`
- `edge-case-hunter` (спека — только если сматчена)

Условный:
- `acceptance-auditor` — ТОЛЬКО при сматченной спеке.

#### Шаблон промпта субагенту (используй для ВСЕХ агентов)

```
=== ВХОД ===

Режим: <pr | branch>
Что ревьюить: <PR #N | текущая ветка против origin/main>
Команда для diff'а: <gh pr diff <N> | git diff $(git merge-base origin/main HEAD)..HEAD>
Команда для полного файла: <git show origin/pr-<N>:<path> | cat <path>>
Спека: <путь | "не передана">

=== СУЩЕСТВУЮЩИЕ INLINE-КОММЕНТАРИИ В PR (для дедупа с прошлыми прогонами) ===

Это inline-комментарии, уже оставленные в PR на GitHub (review-agent'ом в прошлые прогоны или Алиханом). НЕ путать с git-коммитами.

<блок EXISTING_COMMENTS из Шага 1.3, формат: "<path>:<line> — <первая строка body>". Если пусто — "inline-комментариев в PR пока нет, это первый прогон ревью".>

ЗАПРЕЩЕНО выдавать findings, которые буквально совпадают с уже отмеченным в существующих inline-комментариях (тот же path, та же line, та же суть). Если ты считаешь, что прежний комментарий устарел — НЕ выдавай finding, оркестратор разберётся отдельно.

=== ANTI-LAZINESS REQUIREMENTS (ОБЯЗАТЕЛЬНО) ===

Это production code. Пропущенный finding = баг в проде или потерянное время команды. Высокие ставки.

1. **Полнота входа.** Прочитай ВСЕ файлы, которые требует твой базовый промпт (см. .claude/agents/<твоё имя>.md). Не «релевантные», не grep — целиком. Без shortcut'ов.

2. **Покрытие diff'а.** Каждый файл из diff'а либо проверен, либо явно помечен с причиной пропуска (generated / запрещён к чтению по специфике твоей роли — см. свой базовый промпт). Если файл проигнорирован молча — finding потерян.

3. **Покрытие чеклиста / категорий.** Каждый буллет / категория / 11-точечный обход / сверка со спекой — пройден против каждого применимого файла из diff'а. Если категория не применима к файлу — явно зафиксируй: «категория X к файлу Y не применима потому что Z». Молчание = пропуск = баг.

4. **Comprehensive findings.** Выписывай ВСЕ findings, включая `minor`/`nitpick`. Фильтрация — задача оркестратора. Сглаживание «может by design» / «вряд ли случится» — НЕ твоя работа. Лучше выпиши лишнее, чем пропусти.

5. **Конкретность.** Каждое утверждение — с `path:line` и `rationale`. ЗАПРЕЩЕНЫ фразы без конкретики: «остальное в порядке», «стандартно», «проверил — норм», «всё ок», «мелочи опускаю». Каждое утверждение — конкретно.

=== ОБЯЗАТЕЛЬНАЯ СЕКЦИЯ В ОТЧЁТЕ — Self-check ===

В конце отчёта добавь:

```markdown
## Self-check

- Прочитанные обязательные файлы (стандарты/спека/чеклист): [перечислить или "не требовалось"]
- Файлы из diff'а проверены ([N] из [M]): [перечислить все]
- Файлы из diff'а пропущены как generated ([K]): [перечислить]
- Категории / буллеты / разделы пройдены: [перечислить по своему чеклисту]
- Подтверждаю: «Я не пропустил ни одного файла из diff'а (кроме generated) и ни одной категории из своего чеклиста. Я выписал все findings, включая minor/nitpick.»
```

Без этой секции отчёт **невалиден** — оркестратор отправит на повторный прогон. Перед выводом отчёта проверь себя — если хоть один пункт неполон, исправь.
```

Шаблон **универсальный** — одинаковый для всех 4 агентов. Никаких role-specific блоков в промпте оркестратора. Специфика (что именно читать обязательно, как использовать спеку, какие категории проходить) — в базовом промпте каждого агента (`.claude/agents/<имя>.md`), который Claude Code загружает автоматически при спавне субагента.

Поле `Спека` в ВХОДЕ передаётся **всегда** (как сырой параметр):
- Спека сматчена с PR (Шаг 1.5) → `Спека: <путь>`.
- Спеки нет / матчинг низкий → `Спека: не передана`.

Что делать со спекой — решает каждый агент сам по своему базовому промпту. Оркестратор не фильтрует получателей.

### Шаг 3 — Дедуп и сборка

1. Собери findings со всех субагентов в один список.
2. **Валидация Self-check.** Если у субагента секции `## Self-check` нет, или в ней явно пропущен файл / категория без обоснования — сообщи в отчёте «<агент>: невалидный self-check, возможны пропуски». В крайнем случае — повторный спавн этого агента.
3. **Дедуп с EXISTING_COMMENTS.** Для каждого finding'а проверь: есть ли в `EXISTING_COMMENTS` коммент с тем же `(path, line)` и близкой сутью? Если да — выкинь finding (это повтор от прошлого прогона; если коммент устарел — Алихан reclassify в Шаге 8).
4. **Дедуп между субагентами.** Одинаковый `(path, line, нормализованный title)` — одна запись. Нормализация: `trim` + `lowercase` + удаление trailing-пунктуации. В `sources` — все агенты. Severity — максимум.
5. **Семантический дедуп — best-effort.** Очевидно одинаковая суть в разных формулировках на одной строке — мержь. Сомнение — оставь раздельно, пометь в Сводке.
6. **Cross-validation:** findings от 2+ агентов — `**[cross-validated]**` в заголовке.
7. **Сортировка** внутри triage-секций: severity (blocker → nitpick) → cross-validated в начале своей группы.

### Шаг 4 — Triage

Каждое finding → одна категория:
- **fix-now** (default) — нужно исправить в этом PR.
- **clarify** — требует уточнения от Алихана (ambiguity).
- **defer-to-issue** — попадает в один из 5 критериев. Обязательно укажи **под какой**.

Спецслучаи:
- `regression`-finding от acceptance-auditor — всегда `fix-now`.
- `not-implemented [out-of-PR-scope]` от acceptance-auditor — автоматически `defer-to-issue` (критерий 4), без `[требует одобрения]`.

### Шаг 5 — Self-check defer'ов

Для каждого предложенного `defer-to-issue`:
- Критерий не выполнен буквально, severity `minor`/`nitpick` → переклассифицируй в `fix-now` молча.
- Критерий не выполнен буквально, severity `blocker`/`major` → оставь в `defer-to-issue` с пометкой `[не прошёл self-check, требует одобрения]`.
- Критерий выполнен, severity `blocker`/`major` → пометка `[требует одобрения]`.
- Критерий выполнен, severity `minor`/`nitpick` → defer без пометки.

При сомнении — `fix-now`.

### Шаг 6 — Self-check кандидатов в стандарты

Пройди **каждого** кандидата против 5 критериев из секции «Кандидаты в стандарты — СТРОГИЙ ФИЛЬТР»:

1. Паттерн (≥3 примера)? Если subagent не подкрепил числом — сам сделай быстрый `grep`. Не подтвердилось — отбрось.
2. Severity ≥ `major`? Нет — отбрось.
3. Конкретно и проверяемо? Нет — отбрось.
4. Одно предложение, без оговорок? Нет — отбрось.
5. Не дублирует существующее? `grep -ri` по `docs/standards/`. Дубликат — отбрось.

Прошёл все 5 — оставляешь, идёт в отчёт. Иначе — отбрасываешь молча.

### Шаг 8 — HALT, диалог

После вывода отчёта:

#### 8.1 Жди решения Алихана

- По каждому `clarify` — его ответ.
- По каждому `defer-to-issue [требует одобрения]` — confirm / reject / reclassify в `fix-now`.
- Опционально по любому `fix-now` — отдельные правки приоритета или reclassify.

Без явного решения по `[требует одобрения]` — по умолчанию `fix-now`.

#### 8.2 Сбор follow-up через `/interview`

Применяй `/interview` (`.claude/commands/interview.md`) — компактные вопросы с 2-4 вариантами.

**Inline-комменты в PR.**
- `blocker`/`major`-fix-now — постим автоматически.
- `minor`/`nitpick`-fix-now — отдельный вопрос: «постить minor/nitpick тоже? Yes / No / только конкретные». Если minor+nitpick = 0 — вопрос пропускается.

**GitHub issues для одобренных defer-to-issue** — `Yes` / `No` / `Skip`.
- Для `blocker`/`major`-defer'ов ответ `Other` обрабатывается как «переспросить».
- Для `minor`/`nitpick`-defer'ов `Other` = «отбросить».

**Кандидаты в стандарты.** Если кандидатов = 0 — пропусти. Иначе по каждому: «в `<тематический-стандарт>.md` / в Hard rules / в новый стандарт-файл / отбросить».

#### 8.3 Применение действий

**Свежий commit_id** для inline-комментов:
```bash
COMMIT_ID=$(gh pr view <N> --json headRefOid --jq .headRefOid)
```
Берётся ОДИН РАЗ перед началом постинга.

**Inline-комменты с конкретной строкой.** Body **обязан** начинаться с маркера `[🤖 review-agent]` (см. секцию «Маркировка агентских комментов»):

```bash
INLINE_BODY=$(cat <<EOF
[🤖 review-agent]

<rationale>

**Fix:** <fix>
EOF
)
gh api --method POST repos/$REPO/pulls/<N>/comments \
  -f body="$INLINE_BODY" \
  -f path=<path> \
  -f commit_id=$COMMIT_ID \
  -F line=<line> \
  -f side=RIGHT
```
- **Важно:** `-F line=<int>` (uppercase F) — GitHub API требует integer для `line`. С `-f` (lowercase) `gh` отправит строку и API вернёт 422 (`"line" is not an integer`).
- `side=RIGHT` для finding'а на новой/изменённой строке. На удалённой — `side=LEFT`.
- При 422 на `commit_id` (commit устарел) — рефетч `headRefOid` и retry один раз. Если снова 422 — пометка в финальном отчёте.

**Findings без конкретной строки** (`<line>` = «не реализовано», «функция целиком») — группируй в один общий PR-коммент через `gh pr comment`. Маркер `[🤖 review-agent]` в первой строке тоже обязателен:

```bash
GENERAL_BODY=$(cat <<EOF
[🤖 review-agent]

## Review findings без конкретной строки

1. [blocker] **<title>** (sources: ...)
   <rationale>
   Fix: <fix>
EOF
)
gh pr comment <N> --body "$GENERAL_BODY"
```

**Issues:**
```bash
ISSUE_BODY=$(cat <<EOF
<rationale>

Link: <discussion-url>
EOF
)
gh issue create --label tech-debt --label <релевантный> \
  --title "<кратко>" --body "$ISSUE_BODY"
```

- На русском. `<discussion-url>` — `html_url` коммента из EXISTING_COMMENTS или URL PR.
- Отсутствующий label — `gh issue create` упадёт. Создай: `gh label create tech-debt --color "ededed" --description "Tech debt for later iteration"` и retry.

**Кандидаты:** правка конкретного `docs/standards/<file>.md`. `review-checklist.md` правится только при новом слое / Hard rule / стандарт-файле.
