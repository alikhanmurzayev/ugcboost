# Brainstorming Session Results

**Facilitator:** Alikhan
**Date:** 2026-05-05

## Session Overview

**Topic:** Roadmap-2 — следующий спринт от текущего состояния (approve креатора в проде) до «100 креаторов подтвердили приход на мероприятие 13–14 мая».

**Goals:**
- Разбить функционал на чанки правильного размера для одного PR/релиза
- Обеспечить параллельную работу бэка и фронта между чанками
- Уметь стартовать рассылку приглашений до полной готовности всего цикла

## Ключевые решения

- **Дедлайн функционала:** 9 мая (мероприятие 13–14 мая, нужен запас).
- **Канал коммуникации с креатором:** только Telegram-бот. TMA не используется.
- **Создание кампаний:** только админ. Бренд-self-service-флоу — будущее, не закладываем.
- **Статусы кампаний:** не вводим. Только `is_deleted` (soft delete). Если нужно отделить «прошедшее vs текущее» — derived от `event_date`.
- **Внешние интеграции (TrustMe, тикетинг):** реализуем тогда, когда реально подключаются — не мокаем заранее.
- **Принцип нарезки чанков:**
  - Чанк = независимый PR с релизом.
  - Бэк: одна ручка = один чанк.
  - Фронт: одна страница = один чанк (страница может потреблять несколько уже зарелиженных бэк-ручек).

## Поток (целевой user-flow)

```
approved → виден в списке → попал в кампанию → получил приглашение в TG-бот →
согласился → подписал договор (TrustMe) → увидел приватную часть ТЗ
─── пауза до Pre-event ───
напоминание → подтвердил приход → получил билет
```

## Деление на волны

- **Волна 1 (релизим первой, она же — старт рассылки):** список креаторов → CRUD кампаний → выбор креаторов в кампанию → рассылка приглашения в TG → приём согласия/отказа.
- **Волна 2 (катится параллельно, пока креаторы отвечают):** TrustMe-подписание → раскрытие приватной части ТЗ → pre-event reminder → подтверждение прихода → билет.

## Уточнения по Волне 2 (важные развилки)

- **NDA не делаем** — слишком трудоёмко, не успеваем. Приватная часть ТЗ остаётся, но без отдельного NDA-флоу.
- **Где креатор видит приватную часть ТЗ:** вариант (c) — приватная часть = часть договора в TrustMe. Креатор видит её при ознакомлении с договором перед подписью.
- **«Согласен» в боте ≠ финальное согласие.** Это «готов посмотреть полные условия и подписать». Финальное согласие = подпись в TrustMe.
- **Раскрытие приватной части как отдельный блок выкинуто** — оно растворяется внутри TrustMe-флоу.
- **Технические механизмы TrustMe-интеграции = TBD.** Не фиксируем «webhook» / «polling» / способ формирования документа. Спецификация на уровне «бэк инициирует подписание» и «бэк фиксирует факт подписи».
