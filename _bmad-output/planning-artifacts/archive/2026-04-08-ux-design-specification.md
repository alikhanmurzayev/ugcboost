# UX Design Specification — UGCBoost

**Author:** Alikhan
**Date:** 2026-04-07

---

## Executive Summary

### Видение проекта

UGCBoost — управляемый UGC-маркетплейс для Казахстана. Три интерфейса: **мобильное приложение креатора** (MVP: Telegram Mini App, будущее: нативное приложение), **веб-кабинет** (бренды + админы, десктоп), **лендинг** (все, SEO). Ключевой UX-дифференциатор — «ноль координации»: бренд утверждает креаторов из ранжированного списка, всё остальное делает платформа.

### Целевые пользователи

| Пользователь | Профиль | Устройство | Контекст использования |
|---|---|---|---|
| **Креатор** | 18-30 лет, микро-блогер (1-50K), Instagram/TikTok. Не техничный, привык к Telegram и соцсетям | Смартфон (мобильное приложение) | На ходу, между съёмками. Быстрые действия: посмотреть кампанию, подать заявку, сдать ссылку |
| **Бренд-менеджер** | 25-35 лет, маркетолог. Работает за ноутбуком, ценит табличные данные и метрики | Десктоп (веб) | В офисе, за столом. Длинные сессии: создание кампании, ревью заявок, приёмка работ |
| **Админ (Айдана)** | Операционный менеджер, время — дефицит. Параллельно ведёт EFW | Десктоп (веб) | Утренние сессии: обработка очередей модерации, мониторинг проблем |

### Ключевые UX-вызовы

1. **Два мира дизайна.** Мобильное приложение (touch, карточки, вертикальный скролл) vs веб-кабинет (полноценный десктоп SPA). Нужна общая визуальная идентичность при разных UX-подходах
2. **Сложный onboarding.** Креатор проходит: заявка → верификация → модерация → договор TrustMe → приложение. Каждый шаг — потенциальный drop-off. UX должен быть настолько простым, чтобы 22-летний блогер не отвалился
3. **Информационная плотность.** Бренд-менеджер работает с таблицами (ранжированные заявки, метрики, статусы). Нужен баланс между полнотой данных и читаемостью
4. **Очередной модерации.** Админ обрабатывает 20+ заявок за сессию. UX должен минимизировать клики: показать метрики, ссылки на профиль, кнопки одобрить/отклонить — всё в одном месте

### Возможности для UX

1. **Мгновенный запуск.** Приложение должно открываться и быть полезным за секунды. Первое впечатление = доверие
2. **Ранжированный список = вау-момент.** Бренд открывает заявки — видит top-20 с метриками. Это тот момент, когда «руки в Google Sheets» превращается в «15 минут и готово»
3. **Прогресс-индикаторы.** Кампания как pipeline: заявки → одобрено → в работе → сдано → принято. Визуальный прогресс даёт ощущение контроля

## Core User Experience

### Определяющий опыт

**Для креатора:** открыл приложение → увидел кампании → тапнул «Подать заявку» → получил уведомление «Одобрен» → принял заказ (с условиями и NDA) → сдал работу. Весь цикл в одном месте, без переключения контекста.

**Для бренда:** создал кампанию → получил ранжированный список заявок с метриками → одобрил top-20 за 15 минут → отслеживает прогресс на дашборде. Вместо Google Sheets и WhatsApp — одна точка управления.

**Для админа:** утро = очередь модерации. Метрики подтянуты, профили рядом — одобрить/отклонить в один клик. Не нужно открывать Instagram в соседней вкладке.

### Платформенная стратегия

| Платформа | Паттерн взаимодействия | Приоритет |
|---|---|---|
| **Мобильное приложение (креаторы)** | Touch, вертикальный скролл, карточки. Минимум текста, максимум действий. MVP: TMA, будущее: нативное приложение | Мобайл-first |
| **Веб-кабинет (бренды/админы)** | Mouse/keyboard, таблицы, фильтры, формы. shadcn/ui. Sidebar-навигация, data-heavy layout | Десктоп-only |
| **Лендинг** | Статический, SEO, конверсия в заявку креатора | Мобайл-responsive |

### Взаимодействия, которые должны быть мгновенными

- **Подача заявки на кампанию** — один тап + подтверждение ознакомления с ТЗ. Барьер минимальный
- **Принятие заказа (после одобрения)** — осознанный «серьёзный» момент: креатор видит условия NDA, обязательства, тип лицензии, ссылку на рамочный договор. Явное подтверждение. Это не bug, а feature — «припугнуть» намеренно, чтобы отсечь несерьёзных
- **Одобрение/отклонение креатора** — кнопка прямо в карточке, без перехода на отдельную страницу
- **Сдача работы** — вставить ссылку, чекнуть маркировку, отправить. 30 секунд
- **Просмотр статуса** — креатор открывает приложение и сразу видит свои активные кампании и дедлайны

### Критические моменты успеха

| Момент | Роль | Почему критичен |
|---|---|---|
| Первый запуск приложения после одобрения | Креатор | Первое впечатление. Если загрузка > 3с или UI непонятен — потеряли |
| Ранжированный список заявок | Бренд | Вау-момент: «Вот это сервис». Метрики, сортировка, массовое одобрение |
| Уведомление «Одобрена на кампанию» | Креатор | Эмоциональный пик — заказ пришёл без усилий |
| Принятие заказа с условиями | Креатор | Момент серьёзности — NDA, обязательства, лицензия. Отсекает несерьёзных |
| Галерея контента после кампании | Бренд | Итог: стена UGC-контента с метриками. Доказательство ценности |

### Принципы UX

1. **Ноль лишних шагов.** Каждое действие — минимальное количество кликов/тапов. Если можно убрать шаг — убираем
2. **Контекст, не навигация.** Пользователь не должен переходить на другую страницу, чтобы принять решение. Метрики, профиль, действие — в одном месте
3. **Платформо-агностичный UX, свой визуал.** Экраны, флоу и компоненты описываются без привязки к конкретной платформе (TMA / нативное приложение). Визуал — собственная айдентика UGCBoost. MVP реализуется как TMA с нативным поведением Telegram (MainButton, BackButton, haptic), но UX-решения переносимы на нативное приложение
4. **Данные говорят сами.** Бренд-менеджеру нужны числа (ER, подписчики, просмотры) — показываем их компактно и сразу, без раскрытия аккордеонов
5. **Низкий барьер → высокая ответственность.** Подать заявку — легко. Принять заказ — серьёзно (NDA, условия). Осознанная двухступенчатая воронка

## Эмоциональный дизайн

### Эмоциональная карта по ролям

| Роль | Должен чувствовать | Не должен чувствовать |
|---|---|---|
| **Креатор** | «Мне доверяют серьёзные бренды», «Заказы приходят сами», «Всё понятно с первого раза» | Бюрократию, давление, ощущение «меня используют» |
| **Бренд** | «Наконец-то кто-то делает это за меня», «Профессиональный сервис», «Контроль без микроменеджмента» | Хаос, неопределённость, ощущение «дешёвый инструмент» |
| **Админ** | «Всё под контролем», «Быстро разобралась» | Перегруженность, потерянность в данных |

### Эмоции → дизайн-решения

| Эмоция | Как достигается в UI |
|---|---|
| **Доверие** | Чистый UI, профессиональная типографика, юридические элементы (номер договора, статус подписи) видны, но не мешают |
| **Лёгкость** | Минимум полей в формах, прогресс-бары, мгновенный отклик (optimistic updates, haptic feedback) |
| **Контроль** | Статусы видны всегда, фильтры и сортировка, уведомления о каждом изменении |
| **Серьёзность** | Экран принятия заказа — контрастный, с чёткими условиями. Не «мелкий шрифт», а осознанное действие |
| **Достижение** | Уведомление «Одобрена» — яркое, с деталями кампании. Галерея контента — визуальный итог работы |

### Антипаттерны (чего избегаем)

- **Стена текста** на любом экране — особенно на мобайле. Если нужно много текста (договор, ТЗ) — раскрываемые секции
- **Неопределённые статусы** — «Обрабатывается...» без контекста. Всегда конкретно: «На модерации, обычно 1-2 дня»
- **Пустые состояния без подсказки** — «Нет кампаний» → «Пока нет кампаний в вашей категории. Мы уведомим, когда появятся»

## UX-референсы и паттерны

### Референсы по платформам

**Мобильное приложение (креаторы):**
- **Instagram (лента, stories)** — целевая аудитория живёт здесь. Карточный UI, вертикальный скролл, визуальное превью
- **Telegram Mini Apps (Wallet, Fragment)** — референс для MVP-реализации: нативное поведение внутри Telegram

**Веб-кабинет (бренды/админы):**
- **Linear** — data-heavy но чистый. Sidebar + таблицы + быстрые действия. Фильтры без перегрузки
- **Notion-подобные таблицы** — сортировка, фильтры, inline-действия. Привычный паттерн для маркетологов

### Паттерны для адаптации

| Паттерн | Откуда | Как применяем |
|---|---|---|
| **Карточки с действием** | Мобильные приложения | Кампания = карточка с кнопкой «Подать заявку» |
| **Sidebar + content** | Linear, Notion | Навигация веб-кабинета: секции слева, контент справа |
| **Ранжированная таблица** | CRM-системы | Список заявок с метриками, сортировка по score |
| **Pipeline-view** | Trello, Linear | Статусы кампании: заявки → одобрено → в работе → сдано → принято |
| **Empty states с CTA** | Все хорошие SaaS | «Нет кампаний» → «Создайте первую кампанию» с кнопкой |
| **Toast-уведомления** | Любой SPA | Результат действия: «Креатор одобрен», «Кампания опубликована» |

### Антипаттерны

- **Мобайл как урезанный сайт** — приложение должно ощущаться нативным, не веб-страницей в обёртке
- **Модальные окна поверх модальных** — особенно в мобайле. Один уровень вложенности максимум
- **Дашборд ради дашборда** — не нужны графики для 200 пользователей. Конкретные числа и списки полезнее

## Дизайн-система

### Выбор

| Платформа | Система | Обоснование |
|---|---|---|
| **Веб-кабинет** | shadcn/ui + Tailwind CSS | Копируемые компоненты (не npm-зависимость), AI пишет отличный Tailwind, полный контроль над стилями. Таблицы, формы, модалки — всё из коробки |
| **Мобильное приложение** | Tailwind CSS + кастомные компоненты | shadcn/ui заточен под десктоп. На мобайле нужны touch-friendly карточки и компоненты. Общий Tailwind preset обеспечивает единые цвета/шрифты. MVP: TMA, будущее: нативное приложение с тем же UX |
| **Лендинг** | Astro + Tailwind CSS (shared preset) | Статический output, минимум JS, общий визуальный язык |

### Общий Tailwind Preset

Файл `tailwind.preset.ts` в корне монорепы — единый источник правды для:
- Цветовая палитра UGCBoost (primary, secondary, accent, semantic)
- Типографика (шрифты, размеры, line-height)
- Spacing, border-radius, shadows
- Breakpoints

Все три фронтенда наследуют preset, переопределяя только платформо-специфичные значения.

### Кастомизация

- **Бренд UGCBoost** определяет визуал, не Telegram-тема и не дефолтный shadcn
- shadcn/ui компоненты кастомизируются через CSS-переменные и Tailwind-классы — никакого vendor lock-in
- Мобильные компоненты строятся с нуля на Tailwind, используя тот же preset. Переносимы между TMA и нативным приложением

## Определяющий опыт

### Для друга в одном предложении

- **Креатор:** «Открываешь приложение — там заказы от брендов, тапнул — подал заявку»
- **Бренд:** «Создал кампанию — получил список проверенных блогеров с метриками — утвердил за 15 минут»

### Ментальная модель пользователей

| Роль | Привычный способ | Ожидание от UGCBoost |
|---|---|---|
| **Креатор** | Ищет заказы через DM, Threads, знакомых. Нестабильно, унизительно | «Лента заказов» — как лента вакансий, но в телефоне. Заказы приходят ко мне |
| **Бренд** | Google Sheets + WhatsApp + ручной поиск в Instagram. 60% рабочего времени | «CRM для UGC» — создал задачу, получил исполнителей с рейтингом, отследил результат |
| **Админ** | Тот же Sheets + WhatsApp, но за обе стороны | «Операционная панель» — очереди, статусы, кнопки действий |

### Механика ключевых взаимодействий

**1. Креатор подаёт заявку (мобайл)**
- Инициация: карточка кампании в ленте → тап → полный бриф
- Действие: «Подать заявку» → подтверждение ознакомления с ТЗ
- Обратная связь: haptic + экран «Заявка отправлена, ожидайте»
- Завершение: статус в «Мои кампании»

**2. Креатор принимает заказ (мобайл)**
- Инициация: push «Вас одобрили на кампанию X»
- Действие: экран с условиями (NDA, обязательства, лицензия, номер договора) → явное подтверждение
- Обратная связь: haptic + «Заказ принят, дедлайн: [дата]»
- Завершение: кампания в «Активные» с таймером до дедлайна

**3. Бренд ревьюит заявки (веб)**
- Инициация: badge «12 новых заявок» на кампании
- Действие: таблица с метриками (ER, подписчики, просмотры), чекбоксы, «Одобрить выбранных»
- Обратная связь: toast «Одобрено 8 креаторов», счётчик обновился
- Завершение: pipeline-view обновляется в реальном времени

**4. Админ модерирует креатора (веб)**
- Инициация: очередь с badge-счётчиком
- Действие: карточка с метриками LiveDune + ссылки на профили + кнопки «Одобрить / Отклонить»
- Обратная связь: toast + автопереход к следующему в очереди
- Завершение: счётчик очереди -1

## Визуальная основа

### Цветовая система (на основе ugcboost.kz)

| Токен | Цвет | Применение |
|---|---|---|
| **Primary** | Hot pink (~#FF2D8A) | CTA-кнопки, акценты, активные элементы |
| **Primary Light** | Светло-розовый фон | Секции-блоки, hover-состояния |
| **Background** | Белый (#FFFFFF) | Основной фон |
| **Surface** | Светло-серый | Карточки, формы, альтернативные секции |
| **Text Primary** | Тёмный/чёрный | Заголовки |
| **Text Secondary** | Серый | Описания, подписи |
| **Success / Warning / Error** | Зелёный / Жёлтый / Красный | Статусы (одобрен / на модерации / отклонён) |

*Точные hex-значения извлекаются из CSS лендинга при имплементации. Living document — обновляется по ходу работы.*

### Типографика

- **Шрифт:** современный sans-serif (тот же, что на лендинге — извлечь при имплементации)
- **Заголовки:** bold, крупные, чёрные
- **Body:** regular, серый, хорошая читаемость
- **Стиль:** чистый, профессиональный — подходит для бизнес-платформы

### Spacing & Layout

- **Подход:** просторный, с воздухом — не плотный дашборд. Лендинг задаёт тон
- **Базовый модуль:** 4px (Tailwind default)
- **Карточки:** скруглённые углы, лёгкие тени — паттерн с лендинга
- **Мобайл:** вертикальный стек карточек
- **Десктоп:** sidebar + content area, таблицы с достаточным padding

### Accessibility

- Контраст: pink на белом фоне — проверить при имплементации (hot pink может не пройти WCAG AA для мелкого текста). Использовать pink только для кнопок и крупных элементов
- Размер touch-targets: минимум 44x44px на мобайле
- Focus-states: видимые для keyboard navigation в веб-кабинете

## Design Direction Decision

### Исследованные направления

Было исследовано 12 вариаций дизайна в двух раундах (v1: 6 концептуально разных направлений, v2: 6 вариаций победителя). Мокапы доступны: `ux-design-directions.html`.

**Раунд 1 — концептуальные направления:**
1. Modern Card — карточки с градиентами → фидбек: нужны лого брендов
2. Dark Premium — тёмная тема + розовые акценты → отклонено (ассоциации с казино)
3. Soft Warm — пастельные тона, иллюстрации → отклонено (инфантильно)
4. Compact List — компактные строки-списки → отклонено (как Excel, не продаёт)
5. Magazine — журнальные карточки с hero-фото → **ПОБЕДИТЕЛЬ**
6. Gradient Bold — градиентный фон + крупный текст → отклонено (тёмная тема)

**Раунд 2 — вариации Magazine:**
1. Hero Classic — hero-фото + лого бренда оверлеем → **ВЫБРАНО для детальной карточки**
2. Logo Centered — большой лого по центру → отклонено (пустая трата места)
3. Full-Bleed + Emoji — полноширинные обложки → хуже Hero Classic, дешёвые шрифты
4. Grid 2× — двухколоночная сетка → **ВЫБРАНО для списка кампаний**
5. Warm Magazine — тёплый фон + аватар → отклонено (не наши цвета, дёшево)
6. Stories + Cards — сторис-лента + карточки → сторис — идея на будущее, не MVP

### Выбранное направление

**Гибрид: Hero Classic + Grid 2×**

- **Список кампаний** → Grid 2× (двухколоночная сетка). При малом количестве брендов (старт) — крупные карточки а-ля Hero Classic. Заголовок страницы с счётчиком: «Кампании — 6 доступных вашей категории»
- **Детальная карточка кампании** → Hero Classic: большое фото кампании (продукт/интерьер/посылка), лого бренда оверлеем, качественные шрифты, розовая CTA
- **Будущее** → горизонтальная лента сторис брендов (Direction 6) — после масштабирования

### Обоснование

| Решение | Почему |
|---------|--------|
| Hero-фото кампании как главный элемент | Креаторы выбирают по красоте продукта. Фото — не декорация, а контент, привлекающий качественных блогеров |
| Лого бренда на каждой карточке | Бренд загружает свой логотип при создании аккаунта. Узнаваемость бренда = доверие |
| Grid 2× для списка | Больше кампаний видно сразу, удобный обзор. При масштабировании — критически важно |
| Заголовок-счётчик | «6 доступных вашей категории» — сразу понятно контекст и масштаб |
| Качественная типографика | Дешёвые шрифты = дешёвое восприятие платформы. Профессиональный sans-serif из брендинга ugcboost.kz |
| Только розовые CTA | Фирменный цвет, консистентность. Чёрные/серые кнопки выглядят чужеродно |

### Подход к реализации

**Адаптивная плотность:**
- Старт (мало брендов): крупные Hero-карточки на полную ширину → эффект «премиум-каталога»
- Рост (много кампаний): Grid 2× → эффект «Instagram Explore для заказов»
- Переключение автоматическое по количеству доступных кампаний

**Контент карточки кампании:**
- Hero-фото: то, что бренд хочет рекламировать (интерьер ресторана, продукт, посылка)
- Лого бренда: загружается при создании аккаунта, отображается оверлеем или рядом
- Мета-информация: категория, бюджет, дедлайн — компактно, с эмодзи
- CTA: розовая кнопка «Подать заявку»

**Навигация:**
- Заголовок страницы всегда виден сверху (какой экран открыт)
- Счётчик доступных кампаний/категорий в заголовке

## User Journey Flows

*Визуализация: `ux-user-journeys.html`*

### Флоу 1: Креатор — от заявки до сдачи работы

**Онбординг (лендинг → приложение):**

| Шаг | Действие | Примечание |
|-----|----------|------------|
| 1. Заявка | Форма на лендинге: ФИО, ИИН, соцсети, категория, 4 чекбокса ПД | Валидация на клиенте |
| 2. Telegram-бот | Нажимает «Старт» для подписки на уведомления | Fallback: WhatsApp вручную |
| 3. Верификация + автомодерация | **Параллельно:** верификация аккаунта + LiveDune подтягивает метрики | Параллельный запуск сокращает ожидание |
| 4. Ручная модерация | Качество контента, визуал, соответствие категории | Модератор видит метрики из шага 3 |
| 5. Договор TrustMe | Рамочный договор: NDA, обязательства, лицензия. SMS-код | Юридически обязывающий |
| 6. Доступ | Приложение открыто | |

**Рабочий цикл (кампании → сдача):**

| Шаг | Действие | Примечание |
|-----|----------|------------|
| 1. Лента кампаний | Grid 2× / Hero-карточки в категории | Пустое состояние: «Пока нет кампаний в вашей категории. Мы уведомим вас» |
| 2. Бриф | Фото продукта, ТЗ, дедлайн, бюджет | Hero-фото — ключевой элемент |
| 3. Подача заявки | Один тап + подтверждение ознакомления с ТЗ | Минимальный барьер |
| 4. Ожидание | Статус в «Мои заказы» | |
| 5. Одобрение | Push «Вас одобрили!» → wow-экран с деталями кампании + фото продукта + название бренда | Эмоциональный пик — не просто текст, а визуальный wow-moment |
| 6. Принятие заказа | Экран с условиями: NDA, обязательства, лицензия, № договора → явное подтверждение | Осознанный «серьёзный» момент |
| 7. Выполнение | Напоминания: 2нед → 1нед → 3дн → 1день | Один scheduler, проверяет дедлайны ежедневно |
| 8. Сдача работ | Ссылки + чеклист маркировки рекламы | Валидация формата URL, проверка дубликатов |

### Флоу 2: Бренд — от кампании до результатов

| Шаг | Действие | Примечание |
|-----|----------|------------|
| 0. Аккаунт | Команда UGCBoost создаёт аккаунт + загружает лого и фото бренда | MVP: без саморегистрации. Контроль качества фото |
| 1. Создание кампании | Выбор типа (продукт / услуга / ивент) → адаптированный stepper: ТЗ, категория, лимит, дедлайн, маркировка, **фото кампании** | Тип определяет поля брифа. Фото обязательно |
| 2. Модерация | UGCBoost проверяет: правила, закон, маркировка, качество ТЗ | Отклонение с конкретными замечаниями |
| 3. Публикация | Кампания видна креаторам в категории | Email-уведомление бренду |
| 4. Ранжированный список | Top-20/50 с метриками: ER, подписчики, просмотры, кросс-флаг | Вау-момент: 15 минут вместо 2 недель |
| 5. Одобрение | Массовый выбор + «Одобрить выбранных» | Batch-операция |
| 6. Мониторинг | Дашборд: прогресс-бар, pipeline-view | Заявки → одобрено → в работе → сдано → принято |
| 7. Приёмка работ | Сданные ссылки с **oEmbed-превью** (must-have). Статус маркировки | Нативное превью Instagram/TikTok, не переход по ссылке |
| 8. Результаты | Галерея контента: embed-посты + метрики LiveDune | Визуальный итог кампании |

### Флоу 3: Админ — операционный цикл

**Три очереди, одна сессия:**

| Очередь | Действие | UX-решение |
|---------|----------|------------|
| Модерация креаторов | Карточка: метрики LiveDune + ссылки на профили + история | Одобрить/отклонить + фидбэк → **автоподгрузка** следующей карточки (без page reload) |
| Модерация кампаний | Бриф + правила + маркировка | Одобрить / вернуть с замечаниями |
| Алерты | Просрочки, dispute, перегрузка креаторов | Кросс-кампанийная карта: кто в скольких кампаниях |

### Флоу 4: Отклонение и повторная заявка

Креатор → автомодерация ОК → ручная модерация → **отклонение с конкретной рекомендацией** (не «отказано», а «ваш контент — лайфстайл, рекомендуем эту категорию») → уведомление в Telegram → повторная заявка с новой категорией → модератор видит историю → одобрение.

### Флоу 5: Dispute — невыполнение ТЗ

Работа не по ТЗ → бренд/админ отмечает «не соответствует» + комментарий → статус «требуется доработка» + новый дедлайн → **эскалация через Telegram/WhatsApp** (MVP: внеплатформенная — conscious decision; Growth: in-app тикет-система) → исправление или фиксация инцидента → прогрессивные последствия: предупреждение → снижение рейтинга → бан.

### Паттерны поведения

| Паттерн | Где | Описание |
|---------|-----|----------|
| Одно действие — одна карточка | Модерация, ревью | Информация + кнопки на одном экране |
| Автоподгрузка | Очередь модерации | Следующая карточка без page reload |
| Прогрессивная серьёзность | Заявка → принятие | Лёгкий старт, серьёзный коммит |
| Фидбэк с контекстом | Отклонения | Причина + путь к исправлению |
| Напоминания-лестница | Дедлайны | 2нед → 1нед → 3дн → 1день |
| Pipeline-статусы | Кампания (бренд) | Заявки → одобрено → в работе → сдано → принято |
| Batch-операции | Одобрение креаторов | N чекбоксов → одна кнопка |
| Wow-moment | Одобрение на кампанию | Визуальный экран с деталями, не просто текст |

### Принципы оптимизации

1. **Минимум шагов до ценности** — кампании сразу после входа, заявки сразу после публикации
2. **Feedback на каждом шаге** — статусы, toast, progress indicators
3. **Graceful degradation** — конкретное объяснение + путь к исправлению, не тупик
4. **Batch-операции** — массовые действия везде, где это возможно
5. **Параллельные процессы** — верификация + LiveDune одновременно, не последовательно

### Профиль креатора (мобайл)

Tab «Профиль» в bottom navigation. Содержимое:

| Секция | Содержимое | Действия |
|--------|-----------|----------|
| **Статус аккаунта** | Badge вверху: одобрен / на модерации / отклонён / заблокирован | — |
| **Личные данные** | ФИО, город, категория, ссылки на IG/TT | Редактирование (inline или отдельный экран). Категория — read-only, смена через отдельное действие |
| **Смена категории** | Текущая категория + кнопка «Сменить» | Предупреждение: «Смена запускает повторную модерацию. Доступ к кампаниям будет приостановлен» → подтверждение |
| **Портфолио** | Ссылки на примеры работ (публикации в IG/TT) | Добавить / удалить ссылку |
| **Договоры** | Список подписанных договоров (рамочный + акты) со статусами | Просмотр (ссылка на TrustMe или PDF) |
| **История** | Завершённые кампании (архив) | Просмотр деталей |

### Каталог креаторов (веб-кабинет бренда)

Отдельный раздел в sidebar веб-кабинета бренда. Позволяет просматривать всех одобренных креаторов платформы, не только заявки на конкретную кампанию.

**Layout:** таблица (DataTable) с фильтрами.

| Элемент | Описание |
|---------|----------|
| **Фильтры** | Категория, город, диапазон подписчиков, диапазон ER |
| **Строка креатора** | Аватар, имя, категория, подписчики, ER, средние просмотры, кол-во завершённых кампаний |
| **Профиль креатора** | Sheet/drawer: портфолио, метрики LiveDune, история кампаний на платформе |
| **Действие** | «Добавить в избранное» (Growth: амбассадорство) |

*Отличие от ранжированного списка заявок:* каталог — все креаторы платформы; ранжированный список — только подавшие заявку на конкретную кампанию.

### Q&A по кампаниям

Публичная лента вопросов-ответов внутри каждой кампании. Креаторы задают вопросы по ТЗ, бренд отвечает, все участники видят.

**Мобильное приложение (креатор):**
- Внутри экрана кампании — tab или секция «Вопросы» с badge-счётчиком
- Лента: вопрос (автор, дата) → ответ бренда (или «Ожидает ответа»)
- Кнопка «Задать вопрос» → textarea + отправить
- Без вложенных комментариев — плоская лента Q → A

**Веб-кабинет (бренд):**
- Внутри кампании — tab «Q&A» с badge неотвеченных
- Лента вопросов, неотвеченные сверху
- Inline-ответ: textarea под вопросом → «Ответить»
- Уведомление (email) при новом вопросе

### Стратегия реализации

**Разделение ответственности:**
- **Экраны** (pages/views) компонуют компоненты + определяют layout
- **Компоненты** — переиспользуемые, «тупые» (данные приходят через props)
- **Контейнеры** (умные обёртки) — ModerationQueue управляет подгрузкой, карточка просто рендерит

**Приоритет (MVP-first):**
1. **EmptyState** — нужен с первого дня, пустых экранов будет больше, чем наполненных
2. **CampaignCard** (оба варианта) — без него нет ленты кампаний
3. **ApplyButton + WowApprovalScreen** — core flow креатора
4. **RankedCreatorRow + ModerationQueue/Card** — core flow бренда/админа
5. **StatusBadge + DeadlineTimer + MyActiveCampaign** — статусы и home screen
6. **ContentGallery** — после реализации oEmbed-прокси на бэкенде
7. Остальное — по мере реализации user stories

**Общие правила:**
- Мобильные компоненты — Tailwind CSS, touch-friendly (min 44×44px)
- Веб-компоненты — shadcn/ui как база, кастомизация через Tailwind
- Общий `tailwind.preset.ts` — единые цвета, шрифты, spacing

## UX Consistency Patterns

### Иерархия кнопок

| Уровень | Стиль | Применение |
|---------|-------|-----------|
| **Primary** | Розовый фон (#FF2D8A), белый текст | Основное действие: «Подать заявку», «Одобрить», «Сдать работу» |
| **Secondary** | Белый фон, розовая обводка | Второстепенное: «Отменить», «Назад», фильтры |
| **Destructive** | Красный фон | Опасное/отрицательное: «Забанить», «Отклонить» |
| **Ghost** | Десктоп: текст-ссылка. Мобайл: underlined text или mini-button (ghost слишком незаметен на touch) | Третьестепенное: «Подробнее», «Все кампании» |

Одна primary кнопка на экран. **Исключение:** модерация — «Одобрить» (primary) + «Отклонить» (destructive) как два равнозначных CTA.

На мобайле primary CTA — фиксированная внизу экрана, **прячется при скролле вниз, появляется при скролле вверх** (паттерн Instagram).

### Паттерны обратной связи

| Ситуация | Механизм | Пример |
|----------|----------|--------|
| Успех | Toast (зелёный, 3с) + haptic (мобайл) | «Заявка отправлена» |
| Успех (batch) | Один сводный toast, не по одному | «Одобрено 20 креаторов» |
| Ошибка | Inline-ошибка у поля + toast (красный) | «Ссылка невалидна» |
| Предупреждение | Toast (жёлтый) | «Креатор участвует в 3 кампаниях» |
| Загрузка | Skeleton-placeholder (не spinner). Счётчики в заголовках — сразу, до загрузки деталей | «12 заявок» видно мгновенно, строки подгружаются |
| Пустое состояние | EmptyState с подсказкой + CTA | «Пока нет кампаний. Мы уведомим вас» |
| Подтверждение | Dialog с двумя кнопками | «Одобрить 8 креаторов?» |

### Паттерны форм

- Валидация полей ввода: **на blur** (потеря фокуса)
- Валидация textarea (фидбэк, комментарии): **на submit** — не прерывать набор текста
- Ошибки: inline под полем, красный текст, иконка
- Обязательные поля: звёздочка. Необязательные: подпись «(опционально)»
- Длинные формы (создание кампании): **stepper, максимум 4 шага** — основное (название, тип: продукт/услуга/ивент, категория) → бриф (ТЗ, фото, маркировка — поля адаптируются по типу) → условия (бюджет, лимит, дедлайн) → проверка
- **Автосохранение черновика** — закрытая вкладка не = потерянные данные. Ярлык «Черновик» в списке кампаний, возврат на тот же шаг stepper'а

### Паттерны навигации

**Мобильное приложение:**
- Bottom tab bar: **Кампании | Мои заказы | Профиль**
- Badge на табах: не показывать «0» — пустые табы не должны выглядеть мёртвыми
- Заголовок страницы + счётчик всегда вверху
- Назад — нативная кнопка (BackButton в TMA)

**Веб-кабинет:**
- Sidebar (фиксированный): секции по ролям
- Breadcrumbs на вложенных страницах
- Табы для переключения внутри страницы
- **Keyboard shortcuts для модерации (MVP):** `A` = одобрить, `R` = отклонить, `N` = следующий. Только в контексте очереди модерации, не глобально. Tooltip при наведении на кнопку

### Паттерны статусов

Единая цветовая кодировка + иконки (не только цвет — для различимости на мобайле и accessibility):

## Responsive Design & Accessibility

### Responsive-стратегия

Три отдельных фронтенда, не одно адаптивное приложение:

| Платформа | Стратегия | Breakpoints |
|-----------|----------|-------------|
| **Мобильное приложение** | Mobile-only. 320-428px (стандартные смартфоны) | Не адаптируется под десктоп |
| **Веб-кабинет** | Desktop-first. Минимум 1024px | 1024px / 1280px / 1440px |
| **Лендинг** | Mobile-responsive. Полный спектр | 375px / 768px / 1024px / 1440px |

### Accessibility-стратегия

**Уровень: WCAG 2.1 AA** — не отдельная разработка, а корректное использование HTML и shadcn/ui (accessibility-ready из коробки).

| Требование | Реализация |
|-----------|------------|
| Контраст 4.5:1 | Pink (#FF2D8A) только для кнопок и крупных элементов. Мелкий текст — чёрный/серый |
| Touch targets 44×44px | Все интерактивные элементы на мобайле |
| Focus states | Видимый outline в веб-кабинете |
| Semantic HTML | `<button>`, `<nav>`, `<main>`, `<h1-h3>` — не div-soup |
| ARIA labels | На кастомных компонентах (StatusBadge, CategoryChips) |
| Статусы: цвет + иконка | Не только цвет — для различимости |

### Тестирование

- Автоматическое: axe-core / Lighthouse в CI
- Ручное (MVP): keyboard navigation в веб-кабинете, проверка контраста
- Устройства: iPhone SE (маленький экран), Samsung Galaxy (Android)
