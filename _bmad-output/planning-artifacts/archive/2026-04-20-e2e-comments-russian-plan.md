## Обзор

Переписать заголовочные комментарии четырёх e2e тест-файлов (3 бэкенд + 1 фронтенд) на русском языке в нарративном формате, и зафиксировать новое правило в двух файлах стандартов (`backend-testing-e2e.md`, `frontend-testing-e2e.md`) как локальное исключение из общего правила `naming.md:43` "комментарии на английском". Обычные inline-комментарии внутри тестов остаются на английском.

## Требования

### Must-have (обязательные)

- **REQ-1**: Заголовочные комментарии файлов `backend/e2e/auth/auth_test.go`, `backend/e2e/brand/brand_test.go`, `backend/e2e/audit/audit_test.go`, `frontend/e2e/web/auth.spec.ts` переписаны на русском языке.
- **REQ-2**: Формат комментариев — связный нарратив (рассказ о сценариях и их мотивации), а не перечисление/bullet-list.
- **REQ-3**: Технические детали (HTTP-коды ответов, edge cases, объяснения raw HTTP, security-мотивации вида "no enumeration", имена хелперов setup/cleanup) сохраняются — встраиваются в повествование, а не отбрасываются.
- **REQ-4**: Для Go-файлов сохраняется godoc-конвенция: первое предложение начинается с `Package <имя>` (на русском, например `// Package auth — E2E тесты ...`).
- **REQ-5**: Для TypeScript-файла сохраняется JSDoc-блок `/** ... */`.
- **REQ-6**: Секция "Комментарий в начале файла" в `docs/standards/backend-testing-e2e.md` обновлена: явно указан язык (русский), формат (нарратив), и что это локальное исключение из `naming.md`.
- **REQ-7**: Секция "Комментарий в начале файла" в `docs/standards/frontend-testing-e2e.md` обновлена аналогично, адаптировано под Playwright/JSDoc.
- **REQ-8**: Обычные inline-комментарии внутри тестов (объяснения логики, `Raw HTTP: ...`, `Reload page (simulates F5 ...)` и т.п.) не переписываются — остаются на английском.
- **REQ-9**: Поведение тестов не меняется: логика, ассерты, хелперы, данные, cleanup — всё как было. `make test-e2e-backend` и `make test-e2e-frontend` проходят после правок.

### Nice-to-have (желательно)

- **NICE-1**: Единый ритм подачи между четырьмя файлами — одинаковая глубина детализации (один абзац на один `Test*`/`test.describe`), одинаковый тон.
- **NICE-2**: В обновлённых стандартах `backend-testing-e2e.md` и `frontend-testing-e2e.md` привести короткий пример "плохо (bullet-list) → хорошо (нарратив)", чтобы будущему автору e2e-теста не пришлось восстанавливать формат по коду.

### Out of scope

- Логика тестов, хелперы, ассерты, cleanup — не трогаем.
- Unit-тесты (`backend/internal/**/*_test.go`, `frontend/**/*.test.ts`) — не затрагиваются (согласно `backend-testing-unit.md:97-99`, им заголовочный комментарий не нужен).
- `docs/standards/naming.md` — не меняем; локальное исключение оформляется только в e2e-стандартах.
- Любые другие правки в коде (рефакторинг, переименования, упрощения) — отдельные задачи.

### Критерии успеха

1. В рабочем дереве изменены ровно 6 файлов: 4 тестовых + 2 стандарта.
2. `make build-backend` проходит без ошибок.
3. `make lint-backend` проходит без новых предупреждений.
4. `make test-e2e-backend` проходит зелёным.
5. `make test-e2e-frontend` проходит зелёным (или хотя бы `cd frontend/web && npx tsc --noEmit` + `npx eslint src/` — если полноценный e2e-прогон требует запуска всего стека, который сейчас может быть недоступен).
6. Визуальная проверка: заголовочные комментарии в четырёх файлах — на русском, в нарративной форме, без потери технических деталей из оригинала.
7. Правки не закоммичены (оставлены в working tree для ручного ревью Alikhan — согласно проектному правилу "не делать коммиты автоматически").

## Стратегия тестирования

- **Unit-тесты**: не применимы. Задача касается комментариев, а они не исполняются и не тестируются. Unit-тесты бэкенда/фронтенда к этой задаче не относятся.
- **E2E-тесты**: существующие e2e должны продолжать проходить без изменений — они и есть лучшая проверка, что мы ничего не сломали случайно (хотя ломать там, по сути, нечего — мы трогаем только комментарии).
  - Backend: `make test-e2e-backend` (поднимает стек через `docker compose --profile backend`, запускает `go test ./backend/e2e/...`).
  - Frontend: `make test-e2e-frontend` (поднимает backend + web, запускает Playwright). При невозможности поднять стек — минимум `tsc --noEmit` + `eslint src/` на `frontend/web`.
- **Линты**: `make lint-backend` — убедиться, что `golangci-lint` (`errorlint, gocritic, nilerr, paralleltest`) не ругается. Русский текст в godoc-комментарии этими линтерами не проверяется — безопасно.
- **Ручная проверка**: сравнить каждый переписанный заголовок с оригиналом построчно, убедиться что все технические детали (HTTP-коды, security-мотивации, имена хелперов) перенесены в нарратив, а не выброшены.

## Оценка рисков

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Потеря технических деталей при переводе в нарратив (например, забыть упомянуть, что `short password` возвращает 401 без подсказки про формат) | Средняя | На Шаге 7 (кросс-проверка) сравнить каждый переписанный заголовок с оригиналом построчно; пройтись по `Test*` функциям в коде и убедиться, что каждый сценарий (`t.Run`) упомянут в заголовке хотя бы одной смысловой фразой |
| Godoc на русском флагается линтером | Очень низкая | `.golangci.yml` включает только `errorlint, gocritic, nilerr, paralleltest` — ни один не проверяет язык godoc. Проверено на Шаге 8 через `make lint-backend` |
| Нарушение godoc-конвенции (первое предложение без `Package <имя>`) | Низкая | REQ-4 явно требует сохранить префикс. На Шаге 10 ручная проверка `git diff` |
| Рассинхрон `naming.md` и e2e-стандартов (общее правило "на английском" vs новое "заголовок e2e — на русском") | Высокая, если не обновить стандарты | Шаги 1–2 выполняются ДО правок в коде; в обоих e2e-стандартах явно прописывается "локальное исключение из `naming.md`" |
| Разный стиль между четырьмя файлами (где-то три строки, где-то десять) | Средняя | Шаг 7 — кросс-проверка. Плюс договор на Шагах 1–2 в стандарте ("по одному абзацу на `Test*`") даёт якорь |
| JSDoc-блок ломает подсветку/парсинг TypeScript | Очень низкая | JSDoc `/** ... */` — стандартный TS-комментарий, содержимое произвольное. Шаг 9 (`tsc --noEmit`) это ловит |
| Случайно изменилась логика теста при редактировании комментария | Низкая | Шаг 10 — визуальный `git diff` подтверждает, что изменения только в комментарных блоках |
| Русские кавычки «» в коде могут ломать что-то в грядущих тулах | Очень низкая | Комментарии не обрабатываются тулами; риск умозрительный |

## План отката

Изменения локальные, не затрагивают рантайм — откат тривиален.

1. Если что-то пошло не так до коммита:
   ```bash
   git restore backend/e2e/auth/auth_test.go \
               backend/e2e/brand/brand_test.go \
               backend/e2e/audit/audit_test.go \
               frontend/e2e/web/auth.spec.ts \
               docs/standards/backend-testing-e2e.md \
               docs/standards/frontend-testing-e2e.md
   ```
   Working tree возвращается в исходное состояние.
2. Если правки уже закоммичены (не должны — см. REQ "не делать коммиты автоматически"):
   - Откат через `git revert <hash>` или `git reset --hard HEAD~1` (но только с явного разрешения Alikhan — это destructive).
3. Так как поведение тестов и билда не меняется, никакого runtime-отката/миграции/перезапуска инфраструктуры не требуется. Откат — это просто возврат шести файлов в прежнее состояние.
