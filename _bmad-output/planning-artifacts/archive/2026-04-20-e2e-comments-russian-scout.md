## Запрос

Заголовочные комментарии e2e тестов (бэкенд и фронтенд) — те, что в начале файла описывают прогоняемые сценарии, — должны быть **на русском** и в **нарративном формате** (связный рассказ о сценариях, а не сухое перечисление). Обычные комментарии внутри кода (объяснения логики, inline-заметки) — остаются **на английском**.

## Паттерны реализации

### Backend: godoc-форма заголовка

Go-конвенция: первое предложение godoc-комментария пакета начинается с `Package <имя>`. Это важно для `go doc` и читателя кода. При переводе на русский сохраняем префикс:

```go
// Package auth — E2E тесты HTTP-поверхности /auth/*.
//
// TestLogin проверяет POST /auth/login во всех задокументированных ответах.
// Пустой email отправляется сырым HTTP (openapi_types.Email в сгенерированном
// клиенте не пропускает пустую строку), чтобы дойти до серверной валидации
// и получить 422. Несуществующий аккаунт и неправильный пароль отвечают 401
// одинаково — сервер не должен разглашать факт существования пользователя.
// Короткий пароль возвращает тот же 401, чтобы не выдать формат подсказкой.
// Email нормализуется: пробелы и регистр убираются перед поиском. Успешный
// логин отдаёт access token, payload пользователя и HttpOnly-cookie с
// refresh_token.
//
// TestRefresh ...
// ...
//
// Каждый тест сеет своих пользователей через testutil.SetupAdmin /
// SetupAdminClient — они регистрируют cleanup через POST /test/cleanup-entity,
// и после теста данные удаляются (если E2E_CLEANUP=true, это дефолт).
package auth
```

Паттерн: godoc-префикс + короткая шапка о пакете + по одному абзацу на каждый `Test*` — связный рассказ, а не буллеты с кодами ответов.

### Backend: формат нарратива для каждого `Test*`

Плохо (текущее состояние — bullet-list):
```
TestLogin — every documented response of POST /auth/login: empty email
(422, raw), non-existent account (401, no enumeration), wrong password
(401), short password (401, no info leak), ...
```

Хорошо (нарратив):
```
TestLogin проверяет POST /auth/login во всех задокументированных ответах.
Пустой email отправляется сырым HTTP, потому что openapi_types.Email в
сгенерированном клиенте не пропускает пустую строку — так мы доходим до
серверной валидации и получаем 422. Несуществующий аккаунт и неправильный
пароль отвечают 401 одинаково: это защита от перечисления учёток. ...
```

Детали (коды ответов, конкретные защиты, граничные условия) остаются — меняется только формат подачи: от списка → к абзацу с логическими связками (потому что, так мы, при этом, и наконец).

### Frontend: JSDoc на русском, нарратив

```typescript
/**
 * Auth flow E2E — веб-приложение.
 *
 * Тесты проходят полный цикл аутентификации одного администратора. В
 * beforeAll сеем тестового пользователя через POST /test/seed-user, а в
 * afterAll удаляем его (если E2E_CLEANUP != "false").
 *
 * Happy path: пользователь вводит правильный email и пароль, попадает на
 * дашборд, в сайдбаре видит свой email, роль «Админ» и полный набор
 * навигационных ссылок (главная, бренды, аудит) — потому что у админа
 * доступ ко всему. Затем проверяем неправильный пароль: остаёмся на /login
 * и показываем ошибку «Неверный email или пароль» без утечки того, какой
 * именно компонент ошибся. ...
 */
```

Паттерн: связный рассказ о том, что и почему проверяется, вместо нумерованного `Steps: 1. … 2. …`.

## Риски и соображения

- **Godoc на русском**: валиден для Go-компилятора и не конфликтует с текущим `.golangci.yml` (включены только `errorlint, gocritic, nilerr, paralleltest` — правил о языке godoc нет). Для тестового пакета приемлемо, производственный godoc никто не смотрит.
- **Потеря технических деталей при переводе**: текущие комментарии содержат нетривиальные факты (коды ответов, пояснения почему используется raw HTTP, почему не сравниваем access tokens и т.д.). Переписывая в нарратив, **не выкидывать** эти детали — встроить их в повествование.
- **Конфликт со стандартом `naming.md`**: локальное исключение явно прописывается в обоих e2e-стандартах со ссылкой на общее правило. Без такой фиксации следующий разработчик увидит рассогласование.
- **Консистентность между файлами**: после перевода четыре заголовочных блока должны быть в одном стиле — одинаковая глубина детализации, одинаковый тон. Переписывать все четыре за один проход.
- **JSDoc-формат на фронте**: `/** */` сохраняем (это стандартный комментарий-блок TypeScript), просто содержимое переписываем.
- **Ничего не ломается в рантайме**: комментарии не исполняются, изменения безопасны. Тесты после правок должны проходить так же, как до них.

### Ключевые решения

- **Godoc-форма сохраняется на русском** — `// Package auth — ...` вместо переезда на английский заголовок + русское тело. Проще, короче, не ломает godoc-конвенцию.
- **Стандарт обновляется прямо сейчас** — не потом. Иначе изменения в коде будут противоречить действующему `naming.md:43`.
- **Один абзац на один `Test*`** — единообразный ритм подачи, легко сканировать глазами.

### Альтернативы (отклонены)

- *Держать godoc на английском, рядом русский блок комментариев* — дублирование, сложнее поддерживать.
- *Не обновлять стандарты, просто переписать комментарии* — создаст рассинхрон с `naming.md`, следующий разработчик будет путаться.
- *Оставить формат bullet-list, только перевести на русский* — игнорирует половину запроса ("хочу нарратив, а не просто называние сценариев").
