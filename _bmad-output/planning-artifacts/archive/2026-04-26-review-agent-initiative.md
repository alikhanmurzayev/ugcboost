# Инициатива: Review Agent

## Назначение документа

Меta-фреймворк инициативы по построению отдельного агента-ревьюера для PR'ов UGCBoost. Описывает видение, процессную модель, источники сырья для чеклиста, открытые вопросы и concrete next steps.

**Этот документ + переписка в GitHub-тредах PR'ов = самодостаточный контекст инициативы.** Конкретные находки, обсуждения по конкретным фрагментам кода и принятые по ним решения живут в PR'ах (текущая основа — PR #19), не дублируются здесь.

## Контекст и проблема

В UGCBoost подавляющая часть кода пишется AI-агентами. Если тот же агент, который писал код, ревьюит его — ревью смещённое: агент защищает собственные решения, не ищет в них дыры. Это устойчиво пропускает баги, нарушения стандартов, недотесты, говнокод, дублирование.

Алихан тратит время на ручное ревью одних и тех же классов ошибок раз за разом. Качество растёт медленнее чем хотелось бы. Нужен формализованный, делегируемый процесс.

## Видение

**Ревьюер ≠ кодер.** Два разных агента с разными контекстами и оптикой:

- **Кодер** — пишет код, защищает свои решения (естественная пристрастность).
- **Ревьюер** — приходит холодным, без багажа «почему я так сделал». Ищет дыры, нарушения стандартов, ошибки, паттерны говнокода.

**Триггер:** на старте — вручную Алиханом по запросу. По мере калибровки — slash-команда, label на PR, pre-merge gate.

**Output:** структурированный report с findings (категория, severity, file:line, rationale, actionable fix). Не «всё хорошо/плохо», а конкретный список претензий, которые можно обсудить и пофиксить.

**Loop:** Алихан читает findings → комментирует/уточняет в тредах PR → отдельный агент-кодер (или человек) делает фиксы → revisit если остались open findings.

## Процессная модель (целевая)

```
[Триггер]    Алихан: "review текущую ветку" / label / pre-merge hook
     ↓
[Контекст]   Ревьюер читает: diff, docs/standards/, review-checklist, кодовую
             базу для понимания соседних паттернов, спеку (если есть)
     ↓
[Анализ]     Ревьюер проходит findings по категориям:
             • testing (покрытие, изоляция, моки vs контракты, shadow-DTO, и т.п.)
             • repository / data integrity (БД vs backend ответственности)
             • errors (PII в сообщениях, actionable error.Message, и т.п.)
             • security (логи, валидация, авторизация)
             • code quality (мёртвый код, дубль, говнокод, лишние абстракции)
             • process (артефакты, документация, source-of-truth)
     ↓
[Output]     Structured report (Markdown / JSON / inline-comments через gh):
             категория · severity (blocker/major/minor/nitpick) · file:line
             · rationale · actionable fix
     ↓
[Dialogue]   Алихан комментирует в тредах PR, утверждает / отклоняет findings
     ↓
[Fix]        Кодер (другой агент/человек) делает фиксы по утверждённым findings
     ↓
[Revisit]    При необходимости — повторный прогон ревьюера
```

## Источник сырья для чеклиста

Чеклист ревьюера = объединение трёх источников:

1. **`docs/standards/`** — формализованные правила проекта. Hard rules.
2. **«→ Системно:» аннотации в PR-обсуждениях** — паттерны, найденные в живом ревью. Каждая такая аннотация в моём (Claude) ответе на ревью-коммент = кандидат на пункт чеклиста. Извлекаются через `gh api repos/.../pulls/<N>/comments`, фильтруются по `body contains "→ Системно"`, дедупаются, группируются по категориям.
3. **`_bmad-output/decisions/`** — архитектурные ADR (когда появятся).

**Принцип:** чеклист — живой документ. Каждый новый PR с систематическим ревью обогащает его. Не замораживаем версию — версионируем.

**Текущая база v0:** PR #19 (creator-application-submit) — 31 ревью-тред между Алиханом и Claude, ~21 системная аннотация. Это первая критическая масса. Распределение по категориям видно из тредов и ответов Claude.

## Захваченные правила (preview v0 чеклиста)

Hard rules, уже извлечённые из живого ревью. Список расширяется по мере того,
как новые «→ Системно:» аннотации появляются в PR-обсуждениях. Конечный
чеклист ревьюер-агента собирается из этих правил + `docs/standards/`.

- **Миграции, прогнанные на staging — не редактировать. Никогда.** Любая
  правка уже задеплоенной миграции = новая forward-миграция, не in-place
  edit существующего файла. Причина: миграции применяются one-way — они
  фиксированы в истории staging/prod с момента первого `goose up`. Правка
  старой миграции in-place создаёт расхождение между «состоянием БД на
  staging» и «состоянием в репозитории» — следующий `goose up` на свежей
  БД даст одну схему, на staging — другую. Любые поведенческие правки
  (drop CHECK, drop DEFAULT, ALTER TYPE и т.п.) — отдельной forward-миграцией
  с явным `+goose Up` / `+goose Down`. Применяется ко **всем** миграциям,
  которые могли прогнаться где-то кроме локальной dev-машины. На staging
  все миграции в `backend/migrations/` уже прогонялись.
- **Coverage-gate'ы по N% — относятся ко ВСЕМУ backend-коду в покрываемых
  пакетах, не только к публичной поверхности.** Если awk/grep-фильтр в
  Makefile отсекает lowercase идентификаторы (`$2 ~ /^[A-Z]/` или
  аналог) — это баг gate'а: приватные функции и методы становятся
  невидимыми для проверки и не имеют обязательного покрытия. Источник:
  PR #19, обнаружено что фильтр в `test-unit-backend-coverage` пропускал
  всю приватную поверхность сервисов/репозиториев годами.
- _(другие правила будут добавляться по мере появления)_

## Открытые вопросы (решить до построения v0 агента)

- **Где жить чеклисту:** `docs/standards/review-checklist.md` (стандарт-документ)? `.claude/agents/reviewer.md` (system prompt subagent'а)? Оба (чеклист — данные в docs, system prompt — в .claude, читает чеклист)? Склонение: оба, разделение ответственностей.
- **Какие тулзы у ревьюера:** Read, Bash, Grep — обязательно. Gh для PR-контекста — желательно. Edit/Write — НЕТ (ревьюер не пишет код).
- **Формат output'а:** Markdown report → Алихан читает → пишет inline-comments через GitHub UI? Или ревьюер сразу постит inline-comments через gh API? Второй вариант снижает trение, но требует более жёсткого качества (false positives станут заметнее).
- **Где запускать:** локально через Agent tool (`subagent_type: reviewer`)? В CI как GitHub Action? На старте — локально, потом в CI.
- **Severity-калибровка:** явная шкала (blocker / major / minor / nitpick) с критериями. Нужна для приоритизации.
- **False positives:** механизм пометить «не баг» (комментарием? ignore-файл? inline-аннотация в коде?).
- **Bilingual concern:** комменты ревьюера на русском (CLAUDE.md), но сами findings про код — там переменные/файлы английские. Решить как смешивать.

## Concrete next steps

1. **Закрыть PR #19.** Реализовать фиксы по утверждённым findings (см. треды PR), смержить в main. Все решения уже зафиксированы в GitHub.
2. **Extract v0 чеклиста.** Пройти `gh api repos/.../pulls/19/comments` → filter по `→ Системно` → сгруппировать по категориям (testing/repo/errors/security/quality/process) → выписать в draft `docs/standards/review-checklist.md`. Дедуп, нормализация формулировок.
3. **Решить открытые вопросы выше.** Короткая сессия с Алиханом — выбрать пути по каждому пункту.
4. **Построить v0 ревьюер-агента.** `.claude/agents/reviewer.md` с system prompt'ом, ссылающимся на `docs/standards/` и `docs/standards/review-checklist.md`. Скоуп тулзов: read-only.
5. **Тестовый прогон.** На следующем нетривиальном PR — запустить ревьюер, сверить с глазами Алихана. Найдённые / пропущенные / false positive findings — материал для калибровки.
6. **Iterate.** 2-3 цикла калибровки до приемлемого качества. Параллельно — пополнение чеклиста новыми «→ Системно:» из живых ревью.
