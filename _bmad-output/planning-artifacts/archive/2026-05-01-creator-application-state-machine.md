# Стейт-машина заявки креатора

Living document. Описывает жизненный цикл заявки в скоупе MVP-онбординга: от подачи до одобрения, отклонения или отзыва. Источник истины для всех слоёв (БД, бэк, админка, нотификации). Договор / TrustMe / подпись из этого роадмапа уехали в `campaign-roadmap` — онбординг кончается на `approved`.

## Статус заявки

У заявки одно поле `status`. Значения:

**Активные** (заявка движется по pipeline):

- `verification` — креатор подтверждает владение соцсетями
- `moderation` — заявка на ручной модерации

**Терминальные** (заявка завершена):

- `approved` — заявка одобрена; следующие действия — за campaign-onboarding-roadmap
- `rejected` — отклонена админом
- `withdrawn` — зарезервированный терминал на будущее (креатор-инициированный отзыв); в MVP UI и переходов в него нет

## Уникальность по ИИН

Partial unique index по `iin` — только на активных статусах. В терминальных статусах ИИН не блокирует повторную подачу.

`WHERE status IN ('verification', 'moderation')`

## Переходы

### Прямой ход

| От | К | Триггер |
|----|---|---------|
| `verification` | `moderation` | Auto: креатор подтвердил владение соцсетями (либо webhook от SendPulse, либо ручная верификация админом) |
| `moderation` | `approved` | Manual: админ одобрил заявку — реализуется в chunk 18 (см. roadmap группы 5) |

### Админ-инициированный отклон

Из любого активного статуса (`verification` / `moderation`) админ может перевести заявку в `rejected`.

| От | К | Триггер |
|----|---|---------|
| `verification` | `rejected` | Manual: админ отклонил заявку |
| `moderation` | `rejected` | Manual: админ отклонил заявку |

### Откаты

В v2 откатов нет: `approved` окончательный, обратного перехода в онбординг-флоу не предусмотрено. Восстановление ошибочно отклонённой заявки оформляется бизнес-инструкцией («попросите креатора подать новую заявку с тем же ИИН»), а не переходом из `rejected`.

### Креатор-инициированный отзыв

В MVP не реализован. `withdrawn` остаётся в enum как зарезервированный терминал — переходы будут описаны отдельной инициативой, когда фича появится в скоупе.

## Автомодерация (метрики LiveDune)

Метрики из LiveDune — **атрибут заявки**, не отдельный статус. Фоновый воркер тянет метрики у заявок, у которых их ещё нет, независимо от текущего статуса. В админке явно показываем состояние метрик: «загружаются» / «загружено» / «нет данных». Админ может работать с заявкой и без метрик — это подсказка, не блокер.

## История переходов

Каждый переход пишем в **двух местах**:

1. **Отдельная таблица истории переходов** — для бизнес-анализа (сколько висит на этапе, паттерны откатов, статистика по модераторам).
2. **Общий `audit_logs`** — как любое значимое действие в системе.

## Уведомления креатора

Креатор не имеет UI с состоянием заявки. О каждом значимом переходе он узнаёт из уведомления Telegram-бота — конкретные сообщения подключаются по мере реализации соответствующих чанков (см. roadmap группы 1, 3, 5). На стороне TMA / web кабинета креатора в скоупе MVP нет — `frontend/tma/` остаётся как future-work, не часть текущей итерации.
