# Roadmap: кампании — от создания до подтверждения прихода

Living document. Покрывает путь от `approved`-креатора (терминальная точка онбординга) до билета на мероприятие. Подача заявки → верификация → approved — за границей этого roadmap'а, в `creator-onboarding-roadmap.md`. Обновляется по ходу реализации.

## Дедлайн

Функционал должен работать к **9 мая**. Мероприятие — 13–14 мая, цель — 100 креаторов подтвердили приход.

## Как использовать

- Документ — карта маршрута, не story-spec. Уровень — chunk, не экран и не сущность
- Перед стартом каждого chunk — отдельный `/bmad-quick-dev` в `_bmad-output/implementation-artifacts/`, ссылку добавляем сюда
- Отмечаем прогресс чек-боксами: `[ ]` → `[~]` (в работе) → `[x]` (готово, в main)
- Перед любой реализацией агент обязан полностью загрузить `docs/standards/`

## Архитектурные принципы

- **Telegram-бот — единственный канал коммуникации с креатором.** Преемственно из онбординг-roadmap'а — никакого TMA. Все приглашения, согласия/отказы, подтверждения прихода — через бот (текстом и inline-кнопками)
- **Кампании создаёт только админ.** Бренд-self-service-флоу (бренды сами создают, отправляют на ревью) — будущее, не закладываем сейчас
- **Без статусов кампаний.** Только `is_deleted` (soft delete). Если нужно разделить «прошедшее vs текущее» — derived от `event_date`
- **Внешние интеграции делаем когда подключаются, не мокаем заранее.** TrustMe реализуется в момент, когда нужно реально подписывать договор. Тикетинг — аналогично
- **Принцип нарезки чанков:** одна бэк-ручка = один чанк (один PR). Одна фронт-страница = один чанк (страница потребляет несколько уже зарелиженных бэк-ручек)
- **Параллельная разработка.** Бэк и фронт идут чередующимися PR'ами — как только зарелижено достаточно бэк-ручек под одну фронт-страницу, фронт-агент стартует на ней, бэк параллельно идёт в следующий блок ручек

## Волны выкатки

- **Волна 1** (релизим первой, она же — старт рассылки): список креаторов → CRUD кампаний → выбор креаторов → рассылка приглашения → приём согласия/отказа. Достаточно, чтобы начать рассылку приглашений
- **Волна 2** (катится параллельно, пока креаторы отвечают): TrustMe-подписание → pre-event reminder → подтверждение прихода → билет

## Поток (целевой user-flow)

```
approved → виден в списке → попал в кампанию → получил приглашение в TG-бот →
согласился → подписал договор (TrustMe) → увидел приватную часть ТЗ
─── пауза до Pre-event ───
напоминание → подтвердил приход → получил билет
```

> **Уточнение:** «Согласен» в боте ≠ финальное согласие. Это «готов посмотреть полные условия и подписать». Финальное согласие = подпись в TrustMe, где креатор и видит приватную часть ТЗ впервые.

## Путь чанков

### Группа 1. Список креаторов

Креаторы копятся в БД после chunk 18.5/18.6 онбординг-roadmap'а. Нужно показать их в админке.

- [ ] **1. Бэк-ручка списка креаторов.** Admin-only ручка с фильтрами/пагинацией. Item-shape — то, что уже в БД по креатору. Без LiveDune-аналитики (не подтягиваем сейчас). Audit-log не нужен (read-only). Unit + e2e со всеми полями контракта
- [ ] **2. Фронт-страница списка креаторов.** Перенос упрощённой версии из прототипа Айданы в реальный фронт. RoleGuard(ADMIN), подключение к ручке из chunk 1, без LiveDune-блоков. Unit + e2e

### Группа 2. CRUD кампаний (бэк)

- [ ] **3. Миграция `campaigns` + `POST /campaigns`.** Поля минимальные (id, public_brief, private_brief, event_date, is_deleted, timestamps — финальный набор в спецификации). Audit + e2e. Без статусов
- [ ] **4. `GET /campaigns/{id}`.** Read-only детальная ручка. Все поля кампании
- [ ] **5. `PATCH /campaigns/{id}`.** Обновление полей. Audit + e2e
- [ ] **6. `GET /campaigns`.** Список с пагинацией. По умолчанию `is_deleted=false`
- [ ] **7. `DELETE /campaigns/{id}` (soft).** UPDATE is_deleted=true. Audit + e2e

### Группа 3. Фронт кампаний

- [ ] **8. Страница create/edit кампании.** Форма с публичной + приватной частью ТЗ + датой мероприятия. Потребляет 3 + 4 + 5. Unit + e2e
- [ ] **9. Страница списка кампаний.** Таблица с переходом на детальную из chunk 8, soft-delete действие. Потребляет 6 + 7. Unit + e2e

### Группа 4. Выбор креаторов в кампанию

- [ ] **10. Бэк M2M `campaign_creators`: add/remove креатора в кампанию.** Миграция + ручки. Audit + e2e
- [ ] **11. Фронт-мультиселект креаторов на странице кампании.** UI на странице 8/9. Потребляет 10 + переиспользует список из 1. Unit + e2e

### Группа 5. Рассылка приглашений

- [ ] **12. Бэк: рассылка приглашений через бот.** `POST /campaigns/{id}/notify` — отправляет сообщение с публичной частью ТЗ + что даём + что хотим + inline-кнопки «согласен подписать договор» / «отказаться» всем выбранным креаторам. Audit + e2e
- [ ] **13. Фронт: кнопка «разослать приглашения» на странице кампании.** Подключение к chunk 12. Unit + e2e

### Группа 6. Приём решения от креатора

- [ ] **14. Бэк: приём согласия/отказа от креатора через бот.** Inline-button callback → обновление статуса `invited / agreed / declined` (точные значения — в спеке) в `campaign_creators`. Если согласился — триггерит chunk 16 (TrustMe-инициация). Audit + e2e
- [ ] **15. Фронт: индикатор статуса креатора в кампании.** Бейдж/колонка в UI кампании (приглашён / согласился / отказался). Потребляет 14. Unit + e2e

### Группа 7. Договор через TrustMe

Технические механизмы (формирование документа, способ подписи нашей стороной, способ приёма факта подписи — webhook / polling / иное) — **TBD**, специфицируются при старте чанков.

- [ ] **16. Бэк: инициация подписания договора через TrustMe.** Триггерится автоматически из chunk 14 после согласия креатора. Формирование договора с публичной + приватной частью + наша подпись + отправка креатору на подписание. Audit + e2e
- [ ] **17. Бэк: приём факта подписания договора + автонотификация в бот.** На событие подписи: статус «подписан» + бот шлёт креатору короткое подтверждение. Audit + e2e
- [ ] **18. Фронт: индикатор «договор подписан» в кампании.** Расширение UI из 15. Потребляет 17. Unit + e2e

### Группа 8. Pre-event reminder + подтверждение прихода

- [ ] **19. Бэк: рассылка напоминаний о подтверждении прихода.** `POST /campaigns/{id}/remind` — рассылка через бот всем подписавшим. Inline-кнопки «приду» / «не смогу». Audit + e2e
- [ ] **20. Фронт: кнопка «разослать напоминания» на странице кампании.** Подключение к chunk 19. Unit + e2e
- [ ] **21. Бэк: приём подтверждения прихода от креатора через бот.** Inline-button callback → статус подтверждения. Audit + e2e
- [ ] **22. Фронт: индикатор «подтвердил приход» в кампании.** Расширение UI из 18. Потребляет 21. Unit + e2e

### Группа 9. Билет (TBD)

Не расписываем подробно — пока непонятна механика интеграции с системой тикетинга. На уровне эскиза:

- Генерация уникальной пригласительной ссылки в отдельной системе тикетинга (внешняя для нашего проекта)
- Рассылка ссылок креаторам через бот — только тем, кто подтвердил приход в chunk 21
- Фронт-кнопка «разослать билеты» на странице кампании

Детализируется отдельной brainstorming-сессией / спецификацией при подходе к этой группе.

## Тестирование

Преемственно по стандартам `docs/standards/`:
- **Бэк-ручка:** unit (handler/service/repository, gate ≥80% per-method) → ручная проверка локально (curl + чтение БД) → e2e со всеми полями контракта в строгих ассертах конкретными значениями (не «не пустое»). Каждая mutate-ручка проверяет audit-row через `testutil.AssertAuditEntry`
- **Фронт-страница:** unit (хуки + компоненты) → ручная проверка через Playwright MCP (поднять локально, потыкать) → Playwright e2e spec
