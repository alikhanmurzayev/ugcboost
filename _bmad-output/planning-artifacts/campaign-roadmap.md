# Roadmap: кампании — от создания до подтверждения прихода

Living document. Покрывает путь от `approved`-креатора (терминальная точка онбординга) до билета на мероприятие. Подача заявки → верификация → approved — за границей этого roadmap'а, в `creator-onboarding-roadmap.md`. Обновляется по ходу реализации.

## Дедлайн

Функционал должен работать к **9 мая**. Мероприятие — 13–14 мая, цель — 100 креаторов подтвердили приход.

## Как использовать

- Документ — карта маршрута, не story-spec. Уровень — chunk, не экран и не сущность
- Перед стартом каждого chunk — отдельный `/bmad-quick-dev` в `_bmad-output/implementation-artifacts/`, ссылку добавляем сюда
- Отмечаем прогресс чек-боксами: `[ ]` → `[~]` (в работе) → `[x]` (готово, в main)
- Перед любой реализацией агент обязан полностью загрузить `docs/standards/`

## Архитектурные принципы

- **Telegram-бот — основной канал инициации коммуникации.** Все рассылки идут через бот универсальным сообщением со ссылкой. Сбор решения креатора по приглашению (согласие/отказ) — через TMA-страницу по `tma_url` (открывается из бота по inline-кнопке-ссылке). Inline-callback бота для решений по конкретной кампании в этом roadmap'е не используем. Подтверждение прихода (Группа 8) механика — TBD, специфицируется при подходе к группе
- **Кампании создаёт только админ.** Бренд-self-service-флоу (бренды сами создают, отправляют на ревью) — будущее, не закладываем сейчас
- **Без статусов кампаний.** Только `is_deleted` (soft delete). Если нужно разделить «прошедшее vs текущее» — derived от `event_date`
- **Внешние интеграции делаем когда подключаются, не мокаем заранее.** TrustMe реализуется в момент, когда нужно реально подписывать договор. Тикетинг — аналогично
- **Принцип нарезки чанков:** одна бэк-ручка = один чанк (один PR). Одна фронт-страница = один чанк (страница потребляет несколько уже зарелиженных бэк-ручек)
- **Параллельная разработка.** Бэк и фронт идут чередующимися PR'ами — как только зарелижено достаточно бэк-ручек под одну фронт-страницу, фронт-агент стартует на ней, бэк параллельно идёт в следующий блок ручек
- **Эталоны UI / интерфейс.** UI-паттерны (страницы, формы, фильтры, layout, копирайт, локаторы `data-testid`) для новых chunk'ов берём из `features/creators/`, `features/campaigns/`, `features/creatorApplications/`, `features/auth/`, `features/dashboard/`. Интерфейсная часть `features/brands/` и `features/audit/` — legacy, **не ориентируемся**: не копируем UX, структуру форм и визуал. Ограничение касается только UI — бизнес-логика и API-слой этих фич не запрещены к переиспользованию

## Волны выкатки

- **Волна 1** (релизим первой, она же — старт рассылки): список креаторов → CRUD кампаний → выбор креаторов → рассылка приглашения → приём согласия/отказа. Достаточно, чтобы начать рассылку приглашений
- **Волна 2** (катится параллельно, пока креаторы отвечают): TrustMe-подписание → pre-event reminder → подтверждение прихода → билет

## Поток (целевой user-flow)

```
approved → виден в списке → попал в кампанию → получил приглашение в TG-бот →
открыл TMA, увидел ТЗ → согласился → подписал договор (TrustMe)
─── пауза до Pre-event ───
напоминание → подтвердил приход → получил билет
```

> **Уточнение:** «Согласен» в TMA ≠ финальное согласие. Это «готов подписать договор». Финальное согласие = подпись в TrustMe.

## Путь чанков

### Группа 1. Список креаторов

Креаторы копятся в БД после chunk 18.5/18.6 онбординг-roadmap'а. Нужно показать их в админке.

- [x] **1. Бэк-ручка списка креаторов.** Admin-only ручка с фильтрами/пагинацией. Item-shape — то, что уже в БД по креатору. Без LiveDune-аналитики (не подтягиваем сейчас). Audit-log не нужен (read-only). Unit + e2e со всеми полями контракта
- [x] **2. Фронт-страница списка креаторов.** Перенос упрощённой версии из прототипа Айданы в реальный фронт. RoleGuard(ADMIN), подключение к ручке из chunk 1, без LiveDune-блоков. Unit + e2e

### Группа 2. CRUD кампаний (бэк)

- [x] **3. Миграция `campaigns` + `POST /campaigns`.** Поля: `id, name, tma_url, is_deleted, timestamps`. `tma_url` — секретный лендинг внутри TMA-приложения, в нём живёт ТЗ кампании; в БД лежит только ссылка. `name` уникален среди живых (partial unique index). Audit + e2e. Без статусов
- [x] **4. `GET /campaigns/{id}`.** Read-only детальная ручка. Все поля кампании
- [x] **5. `PATCH /campaigns/{id}`.** Обновление полей. Audit + e2e
- [x] **6. `GET /campaigns`.** Список с пагинацией
- [ ] **7. `DELETE /campaigns/{id}` (soft).** UPDATE is_deleted=true. Audit + e2e

### Группа 3. Фронт кампаний

- [~] **8. Страница create/edit кампании.** Форма из 2 полей: `name` + `tma_url`. ТЗ креатор увидит через TMA-страницу по `tma_url`, не через поля в БД. Потребляет 3 + 4 + 5. Unit + e2e
  - [x] **8a. Создание (`/campaigns/new`).** `_bmad-output/implementation-artifacts/spec-campaign-create-frontend.md`. Готово к ревью на ветке `alikhan/campaign-create-frontend`.
  - [ ] **8b. Детальная + edit (`/campaigns/:id`).** Заменяет `campaign-detail-stub`, добавляет редактирование `name` + `tma_url`. Будет отдельным spec'ом после мержа 8a.
- [ ] **9. Страница списка кампаний.** Таблица с переходом на детальную из chunk 8. Soft-delete-кнопка в UI сначала `disabled` (tooltip «появится позже»); активируется отдельным мини-PR'ом, когда chunk 7 в main. Потребляет 6. Unit + e2e

### Группа 4. Выбор креаторов в кампанию

> Полный design Групп 4–6 — `design-campaign-creator-flow.md`. Каждый chunk ниже реализуется отдельной spec через `bmad-quick-dev` со ссылкой на design.

- [ ] **10. Бэк: миграция `campaign_creators` + add/remove/list.** Новая таблица с state-полем (4 статуса: «Запланирован»/«Приглашён»/«Отказался»/«Согласился»), счётчики и timestamps попыток. Ручки `POST /campaigns/{id}/creators` (батч add), `DELETE /campaigns/{id}/creators/{creator_id}` (только до согласия), `GET /campaigns/{id}/creators` (без пагинации). Audit per-creator + e2e
- [ ] **11. Фронт-мультиселект креаторов на странице кампании.** UI на странице 8/9. Потребляет 10 + переиспользует список из 1. Unit + e2e

### Группа 5. Рассылка приглашений

- [ ] **12. Бэк: рассылка приглашений и ремайндеров через бот.** `POST /campaigns/{id}/notify` (первое/повторное приглашение, переход «Запланирован»/«Отказался» → «Приглашён») + `POST /campaigns/{id}/remind-invitation` (ремайндер для статуса «Приглашён»). Universal-сообщение бота без деталей кампании, тело = ссылка на TMA (`tma_url`). Partial-success на runtime delivery: кому доставили — счётчики/timestamps инкрементируются и пишется audit; кому не доставили — `undelivered: [{...}]` в response 200, без записи. Strict-422 на validation (несовместимый статус хоть одного `creator_id` → весь batch отклоняется до доставки). Дополнительно: PATCH `tma_url` в кампании отклоняется (422), если у любого `campaign_creator` `invited_count > 0`. Audit + e2e
- [ ] **13. Фронт: кнопки «разослать приглашение» / «разослать ремайндер» на странице кампании.** Выбор подмножества `creator_ids[]`, отображение `undelivered` после рассылки. Подключение к chunk 12. Unit + e2e

### Группа 6. Приём решения от креатора

- [ ] **14. Бэк + TMA: приём решения креатора через TMA.** Бэк-ручки `POST /tma/campaigns/{secret_token}/agree` и `/decline` (`secret_token` — последний path-segment у `tma_url`). TMA initData middleware (HMAC-валидация → `creators.telegram_user_id`). TMA-страницы по `secret_token` с хардкодом полного ТЗ + 2 кнопки; UI всегда показывает оба варианта без проверок статуса, реакция диктуется ответом бэка (200 переход / 422 несовместимый статус / 403 креатор не в кампании / 404 кампания soft-deleted). Audit `agreed`/`declined` с `actor_id = NULL`. Триггер TrustMe (chunk 16) — out of scope этого design'а, специфицируется в Группе 7. Audit + e2e
- [ ] **15. Фронт: расширение страницы кампании со статусами и счётчиками.** Колонка статуса (4 значения), счётчики попыток приглашения и ремайндеров, последние timestamps. Потребляет A3 + расширяет UI из 11/13. Unit + e2e

### Группа 7. Договор через TrustMe

Технические механизмы (формирование документа, способ подписи нашей стороной, способ приёма факта подписи — webhook / polling / иное) — **TBD**, специфицируются при старте чанков.

- [ ] **16. Бэк: инициация подписания договора через TrustMe.** Автоматический триггер при переходе `campaign_creators` в «Согласился» (бэк-flow добавляется здесь — в chunk 14 он out of scope). Формирование договора (включая ТЗ, без отдельной приватной части — креатор её уже видел в TMA) + наша подпись + отправка креатору на подписание. Audit + e2e
- [ ] **17. Бэк: приём факта подписания договора + автонотификация в бот.** На событие подписи: статус «подписан» + бот шлёт креатору короткое подтверждение. Audit + e2e

## Тестирование

Преемственно по стандартам `docs/standards/`:
- **Бэк-ручка:** unit (handler/service/repository, gate ≥80% per-method) → **agent-self-check локально** (curl + чтение БД, результаты сразу в чат; **без HALT** — расхождение с ожидаемым = баг реализации, агент сам фиксит код, перезапускает self-check и переходит к e2e в той же сессии. HALT допустим только если всплыла продуктовая развилка, не описанная в спеке) → e2e со всеми полями контракта в строгих ассертах конкретными значениями (не «не пустое»). Каждая mutate-ручка проверяет audit-row через `testutil.AssertAuditEntry`
- **Фронт-страница:** unit (хуки + компоненты) → **agent-self-check через Playwright MCP** (поднять локально, потыкать; результаты в чат; без HALT — расхождение фиксится агентом и перепроверяется до тех пор, пока поведение не совпадёт со спекой; затем сразу Playwright e2e в той же сессии) → Playwright e2e spec
