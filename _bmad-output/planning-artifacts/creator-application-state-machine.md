# Стейт-машина заявки креатора

Living document. Описывает жизненный цикл заявки от подачи до подписания договора (или отклонения/отзыва), легальные переходы и представление статусов для креатора. Источник истины для всех слоёв (БД, бэк, админка, TMA).

## Статус заявки

У заявки одно поле `status`. Значения:

**Активные** (заявка движется по pipeline):

- `verification` — креатор подтверждает владение соцсетями
- `moderation` — заявка на ручной модерации
- `awaiting_contract` — модерация пройдена, готовим договор
- `contract_sent` — договор отправлен, ждём подписи креатора

**Терминальные** (заявка завершена):

- `signed` — договор подписан (заявка одобрена)
- `rejected` — отклонена админом
- `withdrawn` — отозвана креатором

## Уникальность по ИИН

Partial unique index по `iin` — на активных статусах. В терминальных статусах ИИН не блокирует повторную подачу.

`WHERE status IN ('verification', 'moderation', 'awaiting_contract', 'contract_sent')`

## Переходы

### Прямой ход

| От | К | Триггер |
|----|---|---------|
| `verification` | `moderation` | Auto: креатор подтвердил владение соцсетями |
| `moderation` | `awaiting_contract` | Manual: админ одобрил |
| `awaiting_contract` | `contract_sent` | Manual: админ нажал «отправить договор» (мы подписываем со своей стороны, креатору уходит SMS) |
| `contract_sent` | `signed` | Auto: вебхук от TrustMe о подписи креатора |

### Откаты

| От | К | Триггер |
|----|---|---------|
| `awaiting_contract` | `moderation` | Manual: админ передумал одобрять |
| `contract_sent` | `awaiting_contract` | Manual: админ отозвал договор |
| `rejected` | `moderation` | Manual: восстановление случайно отклонённой |

### Админ-инициированный отклон

Из любого активного статуса (`verification` / `moderation` / `awaiting_contract` / `contract_sent`) админ может перевести заявку в `rejected`.

### Креатор-инициированный отзыв

Из любого активного статуса креатор может отозвать заявку — переход в `withdrawn`.

## Автомодерация (метрики LiveDune)

Метрики из LiveDune — **атрибут заявки**, не отдельный статус. Фоновый воркер тянет метрики у заявок, у которых их ещё нет, независимо от текущего статуса. В админке явно показываем состояние метрик: «загружаются» / «загружено» / «нет данных». Админ может работать с заявкой и без метрик — это подсказка, не блокер.

## История переходов

Каждый переход пишем в **двух местах**:

1. **Отдельная таблица истории переходов** — для бизнес-анализа (сколько висит на этапе, паттерны откатов, статистика по модераторам).
2. **Общий `audit_logs`** — как любое значимое действие в системе.

## Видимые состояния для креатора (TMA)

Креатору не показываем все 7 значений. Маппинг 7 → 6 видимых состояний:

| Внутренний статус | Что видит креатор |
|---|---|
| `verification` | «Подтвердите владение соцсетями» |
| `moderation` + `awaiting_contract` | «На рассмотрении» |
| `contract_sent` | «Подпишите договор» |
| `signed` | «Поздравляем, вы успешно прошли все этапы. Скоро здесь появятся приглашения от брендов» |
| `rejected` | «Заявка отклонена» |
| `withdrawn` | «Заявка отозвана» |

`moderation` и `awaiting_contract` схлопнуты — для креатора это один период «на рассмотрении», разница только в наших внутренних действиях.

После того как у креатора появится полноценный личный кабинет (отдельная инициатива), отображение `signed`-состояния как заявки уйдёт — заменится на дашборд с предложениями от брендов.
