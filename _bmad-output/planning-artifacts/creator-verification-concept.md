# Концепция верификации владения соцсетями

Living document. Описывает шаг, на котором заявка из статуса `verification` уходит в `moderation` после подтверждения владения соцсетями. Источник истины для бэка, админки и TMA. Структуру хранения (поля, таблицы) проектируем при реализации соответствующего чанка roadmap.

## Цель шага

Заявка из `verification` уходит в `moderation`, как только у креатора подтверждена хотя бы одна соцсеть из указанных при подаче.

## Способы подтверждения

Способ — атрибут конкретной соцсети заявки (`auto` / `manual`). Платформа выводится из самой соцсети, в способе не дублируется.

- **`auto`** — только для Instagram. Креатор видит в TMA код, копирует, отправляет в DM на наш официальный Instagram. SendPulse-бот ловит сообщение, дёргает webhook бэка → бэк сверяет код, проверяет handle отправителя, помечает соцсеть верифицированной.
- **`manual`** — админ из карточки заявки помечает соцсеть как верифицированную «под свою ответственность». Применяется когда у креатора нет Instagram (только TikTok / Threads), либо когда мы и так уверены в личности по ИИН/телефону.

Для TikTok и Threads автоматической верификации сейчас нет — только `manual`. TikTok через SendPulse требует бизнес-аккаунта в TikTok, чего у нас пока нет.

## Verification-код заявки

- Генерируется при подаче заявки (формат `UGC-NNNNNN`).
- Активен пока заявка в статусе `verification`. После перехода в `moderation` код перестаёт быть constraint'ом и теоретически может позже переиспользоваться новой заявкой в `verification` (мы не блокируем это специально).
- Регенерация не предусмотрена — код стабилен в пределах одной заявки и всегда виден креатору в TMA, пока заявка в `verification`.
- Для уже существующих prod-заявок — бэкфил кодов в миграции при выкатке.
- После успешной верификации соцсети повторное сообщение с тем же кодом — no-op.

## Триггер перехода `verification → moderation`

Авто при первой успешной верификации любой соцсети заявки (`auto` или `manual`). Никакой явной кнопки «отправить на модерацию» — переход побочный эффект первого подтверждения.

## Действия на этом шаге

- **Webhook от SendPulse** — приём IG DM (handle отправителя + текст с кодом), парсинг, поиск заявки, проверка handle, верификация IG-соцсети, авто-переход в `moderation`.
- **Manual verify соцсети админом** — действие из admin-карточки заявки.
- **Reject заявки админом** — на `verification` и на любом активном статусе (по state-machine).
- **Withdraw заявки креатором** — из TMA, доступно на любом активном статусе (по state-machine).
- **Поля верификации в admin detail/list-ручках заявки** — админка читает их через расширение существующих ручек (chunk 7). `verification_code` остаётся внутренним полем backend'а до chunk 11, когда появится creator-detail-ручка для TMA.

## Reject и withdraw

`reject` (админом) и `withdraw` (креатором) появляются на этом шаге впервые — по state-machine они доступны на любом активном статусе, но физически реализуются здесь, когда впервые появляется UI и контекст.

## Очерёдность реализации

Бэк первым: storage и код в detail → webhook → manual verify → reject → withdraw. Только потом фронт-админка (UI actions) и TMA (verification UI, withdraw UI). Декомпозиция на чанки — в `creator-onboarding-roadmap.md`.
