# UGCBoost - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for UGCBoost, decomposing the requirements from the PRD, UX Design, and Architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

**Аккаунты и аутентификация:**

FR1: Креатор может подать заявку через форму на лендинге (веб): ФИО, ИИН, телефон, соцсети, категория, город, адрес, 4 чекбокса согласия ПД
FR1a: После подачи заявки креатор перенаправляется в Telegram-бот для получения уведомлений о результатах модерации
FR1b: После одобрения модерации и подписания рамочного договора (TrustMe) креатор получает доступ к Telegram Mini App
FR2: Креатор может заполнить профиль (ФИО, ИИН, город, ссылки на IG/TT, категория)
FR3: Система проверяет возраст креатора (18+) по ИИН при регистрации
FR4: Креатор может дать раздельное согласие на каждый вид обработки ПД (обработка, передача третьим лицам, трансграничная передача, условия платформы)
FR5: Система верифицирует владение аккаунтом соцсети при онбординге (конкретный метод — OQ-3, на проработке)
FR6: Креатор может редактировать данные своего профиля
FR7: Креатор может добавить ссылки на примеры работ (публикации в IG/TT) в портфолио профиля
FR8: Креатор может видеть текущий статус своего аккаунта (на модерации / одобрён / отклонён / заблокирован)
FR9: Менеджер бренда может войти в веб-кабинет по логину/паролю
FR10: Админ может войти в веб-панель по логину/паролю
FR11: Админ может создать аккаунт бренда и назначить менеджеров
FR12: Бренд может иметь несколько менеджеров с равными правами
FR13: Менеджер бренда и админ могут сбросить пароль

**Модерация креаторов:**

FR14: Админ может просматривать список заявок креаторов на модерацию
FR15: Система подтягивает метрики креатора из LiveDune при модерации (подписчики, ER, просмотры, частота публикаций, фейковые подписчики)
FR16: Админ может одобрить или отклонить креатора с текстовым обоснованием. Блокирующие критерии: токсичный/политический контент, несоответствие категории, неоригинальный контент, накрутка, неактивность >2 нед
FR17: Отклонённый креатор получает фидбэк и может подать повторную заявку
FR18: Админ может заблокировать креатора (бан) с указанием причины
FR19: Креатор может сменить категорию — это инициирует ремодерацию
FR20: Админ может просматривать историю заявок и инцидентов креатора

**Договоры и юридическая инфраструктура:**

FR21: Система инициирует подписание рамочного договора через TrustMe при онбординге креатора
FR22: Креатор может подписать рамочный договор через SMS-подтверждение (TrustMe)
FR23: При принятии заказа креатор видит условия (тип лицензии, срок, территория, обязательства по ТЗ, NDA), явно подтверждает согласие, видит номер рамочного договора и может просмотреть полный текст договора
FR24: При приёмке работы брендом/админом формируется акт приёма-передачи
FR25: Креатор может просматривать список своих подписанных договоров и их статусы

**Кампании:**

FR26: Менеджер бренда может создать кампанию с брифом (ТЗ, инструкция по маркировке рекламы, уровень лицензии прав, дедлайны, бартерные условия). Три типа: продукт / услуга / ивент с адаптированными полями
FR27: Менеджер бренда может сохранить кампанию до публикации и вернуться к редактированию (черновик)
FR28: Админ может модерировать кампании перед публикацией (проверка наличия инструкций по маркировке)
FR29: Креатор может просматривать доступные кампании с фильтрацией по категории
FR30: Креатор может просмотреть полный бриф кампании перед подачей заявки
FR31: Креатор может подать заявку на участие в кампании
FR32: Креатор может отозвать свою заявку на кампанию
FR33: Менеджер бренда может просматривать список заявок на кампанию с сортировкой по метрикам (ранжированный список)
FR34: Менеджер бренда может одобрить или отклонить заявку креатора
FR35: Менеджер бренда может видеть статусы заказов по всем своим кампаниям одновременно
FR36: Менеджер бренда может видеть сводку по кампании (количество заявок, одобренных, сданных, принятых работ)

**Контент-цикл:**

FR37: Креатор может сдать работу (ссылки на контент) и подтвердить соблюдение требований маркировки рекламы при сдаче
FR38: Менеджер бренда может принять или отклонить работу с указанием причины (в т.ч. несоответствие ТЗ)
FR39: При отклонении креатор может доработать и повторно сдать работу
FR40: Система автоматически отмечает заказы с просроченным дедлайном и уведомляет админа и бренда
FR41: Админ может зафиксировать инцидент по заказу в истории креатора
FR42: Креатор может просматривать свои прошлые и завершённые кампании

**Галерея контента кампании:**

FR42a: Менеджер бренда может просматривать галерею всего опубликованного UGC-контента по кампании в виде плитки/сетки. Реализация — через oEmbed/embed-коды соцсетей (Instagram Embed, TikTok Embed)
FR42b: Каждая плитка галереи отображает embed поста из соцсети с нативным превью и метриками. Дополнительные метрики (при наличии данных LiveDune) отображаются рядом
FR42c: Галерея оптимизирована под десктоп (бренды работают за компьютерами)

**Каталог креаторов:**

FR43: Менеджер бренда может просматривать каталог одобренных креаторов
FR44: Менеджер бренда может фильтровать креаторов по категории, городу, метрикам
FR45: Менеджер бренда может просматривать профиль креатора с метриками из LiveDune

**Нотификации:**

FR50: Система отправляет уведомления креаторам через Telegram (статус заявки, новые кампании, дедлайны, результат модерации)
FR51: Система отправляет уведомления менеджерам бренда по email (новые заявки, сданные работы, дедлайны)
FR52: Система отправляет напоминания о приближающихся дедлайнах (2 нед / 1 нед / 3 дня / 1 день) с кнопкой подтверждения — креатор подтверждает, что планирует выполнить
FR52a: Админ может видеть, кто из креаторов не подтвердил напоминание — ранний сигнал проблемы
FR53: Админ может отправить уведомление выбранной группе креаторов (по кампании, по статусу, вручную)

**Аудит и безопасность:**

FR54: Система логирует все изменения сущностей в аудит-лог (кто, что, когда, где, почему, старое/новое значение)
FR55: Каждый пользователь видит только данные, к которым имеет доступ по своей роли
FR56: Бренд не может просматривать или редактировать данные другого бренда
FR57: Система валидирует все входные данные на границе API
FR58: Админ может просматривать аудит-лог

**Администрирование:**

FR59: Админ может видеть операционный дашборд (общее количество креаторов по статусам, активные кампании, просроченные дедлайны)

**Кросс-кампанийная видимость:**

FR60: Менеджер бренда видит флаг участия креатора в других кампаниях при просмотре заявок (настраивается per-brand)
FR61: Админ может просматривать кросс-кампанийную карту (распределение креаторов по кампаниям)

**Growth (отложено из MVP по решению архитектуры):**

FR46: (Growth) Креатор может задать вопрос по кампании в публичном Q&A
FR47: (Growth) Менеджер бренда может отвечать на вопросы в Q&A кампании
FR48: (Growth) Участники кампании могут просматривать все вопросы и ответы
FR49: (Growth) Система уведомляет менеджера бренда о новых вопросах в Q&A

### NonFunctional Requirements

**Безопасность:**

NFR1: Все данные передаются по TLS 1.2+ (шифрование in transit)
NFR2: Разделение доступа по ролям (RBAC): креатор, менеджер бренда, админ — каждый видит только свои данные
NFR3: Аутентификация: Telegram auth для креаторов, логин/пароль + сброс пароля для брендов и админов
NFR4: Защита от OWASP Top 10: injection, XSS, CSRF, broken access control
NFR5: Принцип наименьших привилегий на каждом эндпоинте API
NFR6: Аудит-лог хранится в основной БД системы (отдельные таблицы), отделён от application-логов
NFR7: Шифрование at rest не требуется для MVP (ИИН + ФИО = общедоступные ПД по разъяснению МЦРИАП)

**Производительность:**

NFR8: TMA: загрузка списка кампаний < 3 секунд
NFR9: Веб-кабинет: загрузка каталога креаторов (200 записей) < 2 секунды
NFR10: Уведомления креаторам через Telegram-бот — доставка в течение 1 минуты после события
NFR11: Уведомления брендам по email — доставка в течение 5 минут
NFR12: Система выдерживает 200 активных пользователей одновременно (пик EFW)

**Надёжность и деплой:**

NFR13: Zero-downtime deploys: деплой, миграции и откат в любой момент без даунтайма
NFR14: Бэкапы БД: continuous backup (WAL archiving), хранение минимум 7 дней
NFR15: RTO (время восстановления) ≤ 4 часа
NFR16: Данные юридически значимы (ИИН, статусы договоров) — потеря недопустима

**Observability:**

NFR17: Health check эндпоинт для мониторинга доступности
NFR18: Метрики приложения: response time, error rate, активные пользователи
NFR19: Алертинг при даунтайме или всплеске ошибок
NFR20: Application-логи (stdout/stderr) для дебага и диагностики

**Тестируемость:**

NFR21: 100% покрытие автоматизированными тестами всех user flows
NFR22: Бэкенд: unit-тесты (бизнес-логика, валидация, RBAC), интеграционные тесты (API, БД, внешние интеграции)
NFR23: Фронтенд: автоматизированные UI-тесты для всех интерфейсов (TMA, веб-кабинет, админка)
NFR24: E2E-тесты: полное покрытие всех сценариев (позитивных и негативных)
NFR25: Идемпотентность тестов: работают на непустой БД, уникальные тестовые данные
NFR26: CI < 10 минут, запуск после каждого коммита

**Локализация:**

NFR27: Язык интерфейса: русский
NFR28: Архитектура поддерживает i18n с первого дня (строки в отдельных файлах, не хардкод). Формальный i18n framework — Growth

**Платформы:**

NFR29: Креаторы: Telegram Mini App (мобайл-приоритет, работает и на десктопе)
NFR30: Бренды: веб-кабинет, оптимизирован под десктоп
NFR31: Админы: веб-панель, десктоп

### Additional Requirements

**Из Architecture Decision Document:**

AR1: Монорепа со starter-структурой: Go backend (Chi v5 + pgx + squirrel), React web/tma (Vite + TS + shadcn/ui + TanStack Query + Zustand), Astro landing — определяет Epic 1 Story 1
AR2: Contract-first API — OpenAPI 3.x YAML → codegen для Go (oapi-codegen → Chi handler interface + types) и TS (openapi-typescript → types + fetch client)
AR3: Порты и адаптеры для всех интеграций — каждая внешняя интеграция = интерфейс + реальная реализация + мок. Переключение через env var (*_MOCK=true)
AR4: JWT auth — short-lived access token + refresh в httpOnly cookie, единый формат для всех ролей (поле role в payload)
AR5: Двухслойная авторизация — middleware (грубая проверка роли → 403) + пакет authz (fine-grained проверка владения: CanManageCampaign, CanApproveCreator, CanSubmitWork)
AR6: DB utility layer — internal/dbutil: интерфейс DB (мокается в тестах) + generic-хелперы One[T], Many[T], Val[T], Vals[T], Exec (принимают sq.Sqlizer)
AR7: Background jobs — robfig/cron внутри основного процесса (напоминания о дедлайнах, LiveDune polling, email-рассылки)
AR8: Backwards-compatible миграции БД — goose, SQL-миграции, zero-downtime
AR9: Docker Compose + Dokploy — деплой на VPS, Traefik reverse proxy, auto-SSL
AR10: CI/CD — GitHub Actions: lint → test-unit → build Docker image → deploy staging → E2E tests → (ручной approve) → deploy prod
AR11: Security hardening — Cloudflare (DDoS/WAF, скрытие IP), SSH hardening (ключи, кастомный порт, fail2ban), Docker non-root, secrets в Dokploy env vars
AR12: 3 VPS у KZ-провайдера — prod + staging + backup, rsync daily
AR13: Zero Trust валидация — два уровня: OpenAPI-автоматический (required, типы, enums) + бизнес-валидация в handler (формат ИИН, 18+, URL соцсетей)
AR14: Error handling — domain errors (ErrNotFound, ErrForbidden) → HTTP status + строковые коды (CAMPAIGN_FULL, CREATOR_NOT_FOUND). Локализация ошибок целиком на фронте (Record<string, string>)
AR15: Upload endpoint + хранение изображений (hero-фото кампании, лого бренда) — Docker volume для MVP, S3 → Growth
AR16: Автосохранение черновика кампании — draft-статус в БД + debounced PATCH или localStorage
AR17: oEmbed API proxy — серверный прокси для Instagram Embed API (access token) + TikTok oEmbed (rate limits). Must-have для ContentGallery

**Контекст из Open Questions (влияет на скоупинг сторей):**

OQ-1: TikTok через LiveDune — не подтверждён. Fallback: ручной скрининг TT-креаторов
OQ-2: API TrustMe — доступ не получен. Fallback: ручная отправка через web-интерфейс TrustMe
OQ-3: Метод верификации креаторов — не определён (OAuth / верификационный пост / иное). Решение на этапе имплементации
OQ-7: Таксономия категорий — финальный список не определён

### UX Design Requirements

**Дизайн-система:**

UX-DR1: Цветовая система из ugcboost.kz — Primary hot pink (~#FF2D8A) для CTA, semantic цвета для статусов (зелёный/жёлтый/красный), белый фон, серый surface
UX-DR2: Типографика — современный sans-serif из лендинга, bold заголовки, regular body, профессиональный стиль
UX-DR3: Shared Tailwind preset (tailwind.preset.ts) — единые цвета, шрифты, spacing, border-radius для всех 3 фронтов (web, tma, landing)
UX-DR4: Иерархия кнопок — Primary (розовый фон, белый текст), Secondary (белый фон, розовая обводка), Destructive (красный), Ghost (текст-ссылка). Одна primary кнопка на экран. На мобайле primary CTA фиксирована внизу

**Компоненты мобильного приложения (креаторы):**

UX-DR5: CampaignCard — variant="compact" для Grid 2×, variant="full" для детального просмотра. Hero-фото кампании, лого бренда, мета-информация (категория, бюджет, дедлайн), CTA. Состояния: default, loading, locked (чужая категория)
UX-DR6: StatusBadge — единая цветовая кодировка + иконки (не только цвет): pending (жёлтый ⏳), approved (зелёный ✅), rejected (красный ❌), in-progress (синий 🔵), submitted (фиолетовый 📎), completed (серый ✔️)
UX-DR7: ApplyButton — «Подать заявку» с подтверждением ознакомления с ТЗ. Состояния: default, confirming, submitted, disabled (лимит)
UX-DR8: WowApprovalScreen — full-screen takeover при одобрении на кампанию: фото кампании, лого бренда, анимация. Мост между уведомлением и принятием заказа
UX-DR9: MyActiveCampaign — карточка на home screen: название, DeadlineTimer, статус, кнопка «Сдать работу». Состояния: active, warning, submitted
UX-DR10: DeadlineTimer — обратный отсчёт до дедлайна. Состояния: normal, warning (≤3 дня), critical (≤1 день)
UX-DR11: SubmitWorkForm — сдача работ: ссылки на контент (настраиваемое количество полей по ТЗ + кнопка «+» для бонусного) + чеклист маркировки рекламы. Валидация URL, проверка дубликатов
UX-DR12: EmptyState — пустое состояние с контекстной подсказкой и CTA. Примеры: «Пока нет кампаний в вашей категории. Мы уведомим вас»
UX-DR13: CategoryChips — горизонтальный скролл chips (как YouTube). Состояния: active, locked (полупрозрачный для чужих категорий)
UX-DR14: NotificationBanner — in-app уведомления. Состояния: info, success, warning
UX-DR15: ProfileScreen — tab «Профиль»: статус аккаунта (badge), личные данные (редактирование), смена категории (с предупреждением о ремодерации), портфолио, договоры, история кампаний

**Компоненты веб-кабинета (бренды + админы):**

UX-DR16: RankedCreatorRow — строка в таблице заявок с метриками (подписчики, ER, просмотры) + badge кросс-кампании. Состояния: default, selected, flagged
UX-DR17: ModerationQueue — умная очередь с автоподгрузкой следующей карточки после действия (без page reload). Состояния: loading, active, empty
UX-DR18: ModerationCard — карточка внутри ModerationQueue: метрики LiveDune + ссылки на профили + кнопки одобрить/отклонить с полем фидбэка
UX-DR19: PipelineView — визуализация стадий кампании: заявки → одобрено → в работе → сдано → принято
UX-DR20: ContentGallery — галерея сданного контента с oEmbed-превью (Instagram Embed, TikTok Embed). Серверная зависимость: API-прокси (Instagram access token, TikTok rate limits)
UX-DR21: AlertBanner — алерт о проблемах (просрочки, dispute). Состояния: warning, critical
UX-DR22: CreatorCatalogRow — строка в каталоге: аватар, имя, категория, город, подписчики, ER, просмотры, кол-во завершённых кампаний

**UX-паттерны:**

UX-DR23: Навигация — мобайл: bottom tab bar (Кампании | Мои заказы | Профиль) + badge-счётчики. Веб: sidebar (фиксированный) с секциями по ролям + breadcrumbs
UX-DR24: Форма создания кампании — stepper макс. 4 шага (основное → бриф → условия → проверка) + автосохранение черновика
UX-DR25: Адаптивная плотность ленты кампаний — Hero-карточки при малом количестве, Grid 2× при росте. Переключение автоматическое по количеству

**Accessibility:**

UX-DR26: WCAG 2.1 AA — контраст 4.5:1 (pink только для кнопок/крупных элементов), touch targets 44×44px, focus states в веб-кабинете, semantic HTML, ARIA labels на кастомных компонентах, axe-core/Lighthouse в CI

### Story 1.2: Аутентификация и RBAC

As a **менеджер бренда или админ**,
I want **войти в веб-кабинет по логину/паролю и видеть только свои данные**,
So that **доступ к системе защищён и разделён по ролям**.

**FRs:** FR9, FR10, FR13, FR55, FR57
**ARs:** AR4, AR5, AR13
**UX-DRs:** UX-DR23
**NFRs:** NFR1–5

**Acceptance Criteria:**

**Given** миграция Story 1.2
**When** применяется
**Then** создаётся таблица `users`: id (UUID), email (UNIQUE), password_hash (TEXT NOT NULL), role (ENUM: admin, brand_manager), created_at, updated_at
**And** seed-миграция создаёт админа (email/пароль из env vars)
**And** пароли хэшируются bcrypt cost ≥ 12 (соль встроена в bcrypt, отдельный столбец не нужен)

**Given** пользователь на странице логина
**When** вводит корректный email и пароль
**Then** получает JWT access token (short-lived, 15 мин) в response body
**And** refresh token устанавливается в httpOnly secure cookie
**And** payload JWT содержит: user_id, role, exp

**Given** access token истёк
**When** клиент вызывает POST /api/auth/refresh
**Then** возвращается новый access token
**And** refresh token ротируется (новый cookie)

**Given** пользователь залогинен
**When** вызывает POST /api/auth/logout
**Then** refresh token инвалидируется (cookie удаляется)

**Given** пользователь забыл пароль
**When** POST /api/auth/password-reset-request с email
**Then** генерируется одноразовый токен сброса (срок жизни ≤ 1 час)
**And** (для MVP: токен логируется в stdout; email-отправка — Epic 3)

**Given** валидный токен сброса
**When** POST /api/auth/password-reset с новым паролем
**Then** пароль обновляется (bcrypt cost ≥ 12), токен инвалидируется

**Given** запрос к защищённому эндпоинту
**When** JWT отсутствует или невалиден
**Then** ответ 401 Unauthorized

**Given** запрос к эндпоинту с ограничением по роли
**When** роль пользователя не совпадает (middleware RBAC)
**Then** ответ 403 Forbidden

**Given** middleware пропустил запрос
**When** нужна fine-grained проверка (например, "менеджер этого бренда")
**Then** пакет `authz` проверяет ownership (CanManageCampaign, CanApproveCreator и т.д.)
**And** при нарушении — 403 с domain error

**Given** веб-кабинет
**When** пользователь залогинен
**Then** отображается sidebar с секциями по роли (UX-DR23)
**And** страница логина использует дизайн-систему (shadcn/ui + tailwind preset)

**Given** CORS конфигурация
**When** запрос с фронтенда
**Then** разрешены только домены из whitelist (env var)

**Given** входные данные на POST /api/auth/login
**When** email или пароль не соответствуют формату
**Then** Zero Trust валидация отклоняет до бизнес-логики (AR13)

### Story 1.4: Аудит-лог

As an **админ**,
I want **видеть аудит-лог всех изменений в системе**,
So that **можно отследить кто, когда и что менял для безопасности и compliance**.

**FRs:** FR54, FR58
**NFRs:** NFR6

**Acceptance Criteria:**

**Given** миграция Story 1.4
**When** применяется
**Then** создаётся таблица `audit_logs`: id (UUID), actor_id (FK → users), actor_role, action (TEXT), entity_type (TEXT), entity_id (UUID), old_value (JSONB, nullable), new_value (JSONB, nullable), reason (TEXT, nullable), ip_address (INET), created_at
**And** индексы на (entity_type, entity_id), (actor_id), (created_at)

**Given** бизнес-операция (создание бренда, назначение менеджера, и т.д.)
**When** операция выполняется
**Then** запись аудит-лога создаётся в **той же транзакции** через RepoFactory + WithTx
**And** если транзакция откатывается — аудит-лог тоже откатывается (атомарность)
**And** AuditRepository — часть RepoFactory

**Given** запись аудит-лога
**When** сохраняется
**Then** фиксируется: actor_id, actor_role, action (e.g. "brand.create", "brand_manager.assign"), entity_type, entity_id, old_value (до изменения), new_value (после), IP-адрес, timestamp

**Given** админ залогинен
**When** открывает раздел «Аудит-лог» в веб-панели
**Then** видит таблицу записей с пагинацией (newest first)
**And** может фильтровать по: actor (email), entity_type, action, date range
**And** каждая строка показывает: дата, актор, действие, сущность, diff old/new

**Given** менеджер бренда залогинен
**When** пытается открыть аудит-лог
**Then** ответ 403 Forbidden

**Given** существующие операции из Story 1.2 и 1.3
**When** выполняются после деплоя Story 1.4
**Then** все операции (логин, создание бренда, назначение менеджера) пишут аудит-лог
**And** ретрофит: обновляются хэндлеры Story 1.2–1.3 для записи аудита через RepoFactory

**Given** аудит-лог
**When** пользователь пытается удалить или изменить запись
**Then** операция невозможна (append-only, нет DELETE/UPDATE эндпоинтов)
