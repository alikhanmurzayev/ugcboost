## PRD Analysis

### Functional Requirements (59 FR)

#### Управление аккаунтами и аутентификация (FR1–FR13)

- **FR1:** Креатор может зарегистрироваться через Telegram Mini App
- **FR2:** Креатор может заполнить профиль (ФИО, ИИН, город, ссылки на IG/TT, категория)
- **FR3:** Система проверяет возраст креатора (18+) по ИИН при регистрации
- **FR4:** Креатор может дать раздельное согласие на каждый вид обработки ПД
- **FR5:** Система верифицирует владение аккаунтом соцсети при онбординге (метод — OQ-3)
- **FR6:** Креатор может редактировать данные своего профиля
- **FR7:** Креатор может добавить ссылки на примеры работ в портфолио профиля
- **FR8:** Креатор может видеть текущий статус своего аккаунта
- **FR9:** Менеджер бренда может войти в веб-кабинет по логину/паролю
- **FR10:** Админ может войти в веб-панель по логину/паролю
- **FR11:** Админ может создать аккаунт бренда и назначить менеджеров
- **FR12:** Бренд может иметь несколько менеджеров с равными правами
- **FR13:** Менеджер бренда и админ могут сбросить пароль

#### Модерация креаторов (FR14–FR20)

- **FR14:** Админ может просматривать список заявок на модерацию
- **FR15:** Система подтягивает метрики из LiveDune при модерации
- **FR16:** Админ может одобрить или отклонить креатора с обоснованием
- **FR17:** Отклонённый креатор получает фидбэк и может подать повторно
- **FR18:** Админ может заблокировать креатора (бан) с причиной
- **FR19:** Креатор может сменить категорию → ремодерация
- **FR20:** Админ может просматривать историю заявок и инцидентов

#### Договоры и юридическая инфраструктура (FR21–FR25)

- **FR21:** Система инициирует рамочный договор через TrustMe при онбординге
- **FR22:** Креатор может подписать договор через SMS-подтверждение
- **FR23:** При принятии заказа креатор видит условия (лицензия, NDA, обязательства), подтверждает, видит номер договора, может просмотреть полный текст
- **FR24:** При приёмке работы формируется акт приёма-передачи
- **FR25:** Креатор может просматривать свои договоры и их статусы

#### Кампании (FR26–FR36)

- **FR26:** Менеджер бренда может создать кампанию с брифом (ТЗ, маркировка, лицензия, дедлайны, бартер)
- **FR27:** Менеджер бренда может сохранить кампанию до публикации
- **FR28:** Админ может модерировать кампании перед публикацией
- **FR29:** Креатор может просматривать кампании с фильтрацией по категории
- **FR30:** Креатор может просмотреть полный бриф перед подачей заявки
- **FR31:** Креатор может подать заявку на кампанию
- **FR32:** Креатор может отозвать заявку
- **FR33:** Менеджер бренда может просматривать список заявок с сортировкой
- **FR34:** Менеджер бренда может одобрить или отклонить заявку
- **FR35:** Менеджер бренда может видеть статусы по всем кампаниям одновременно
- **FR36:** Менеджер бренда может видеть сводку по кампании

#### Контент-цикл (FR37–FR42)

- **FR37:** Креатор может сдать работу и подтвердить маркировку рекламы
- **FR38:** Менеджер бренда может принять или отклонить работу с причиной
- **FR39:** При отклонении креатор может доработать и повторно сдать
- **FR40:** Система отмечает просроченные заказы и уведомляет
- **FR41:** Админ может зафиксировать инцидент в истории креатора
- **FR42:** Креатор может просматривать прошлые и завершённые кампании

#### Каталог креаторов (FR43–FR45)

- **FR43:** Менеджер бренда может просматривать каталог одобренных креаторов
- **FR44:** Менеджер бренда может фильтровать по категории, городу, метрикам
- **FR45:** Менеджер бренда может просматривать профиль креатора с метриками

#### Q&A по кампаниям (FR46–FR49)

- **FR46:** Креатор может задать вопрос в публичном Q&A
- **FR47:** Менеджер бренда может отвечать на вопросы
- **FR48:** Участники могут просматривать все вопросы и ответы
- **FR49:** Система уведомляет бренда о новых вопросах

#### Нотификации (FR50–FR53)

- **FR50:** Уведомления креаторам через Telegram
- **FR51:** Уведомления менеджерам бренда по email
- **FR52:** Напоминания о дедлайнах
- **FR53:** Админ может отправить уведомление группе креаторов

#### Аудит и безопасность (FR54–FR58)

- **FR54:** Аудит-лог всех изменений сущностей
- **FR55:** Видимость данных по роли
- **FR56:** Изоляция данных бренда
- **FR57:** Валидация входных данных на границе API
- **FR58:** Админ может просматривать аудит-лог

#### Администрирование (FR59)

- **FR59:** Админ может видеть операционный дашборд

### Non-Functional Requirements (33 NFR)

#### Безопасность (NFR1–NFR7)

- **NFR1:** TLS 1.2+ для всех данных in transit
- **NFR2:** Шифрование at rest — зависит от OQ-15
- **NFR3:** Разделение доступа по ролям
- **NFR4:** Аутентификация: TG auth (креаторы), логин/пароль + сброс (бренды/админы)
- **NFR5:** Защита от OWASP Top 10
- **NFR6:** Принцип наименьших привилегий на каждом эндпоинте
- **NFR7:** Аудит-лог в основной БД (отдельные таблицы)

#### Производительность (NFR8–NFR12)

- **NFR8:** TMA загрузка кампаний < 3 секунд
- **NFR9:** Веб-кабинет загрузка каталога < 2 секунды
- **NFR10:** Telegram-уведомления в течение 1 минуты
- **NFR11:** Email-уведомления в течение 5 минут
- **NFR12:** 200 активных пользователей одновременно

#### Надёжность и деплой (NFR13–NFR16)

- **NFR13:** Zero-downtime deploys
- **NFR14:** Continuous backup (WAL archiving)
- **NFR15:** RTO ≤ 4 часа
- **NFR16:** Данные юридически значимы — потеря недопустима

#### Observability (NFR17–NFR20)

- **NFR17:** Health check эндпоинт
- **NFR18:** Метрики: response time, error rate, активные пользователи
- **NFR19:** Алертинг при даунтайме или всплеске ошибок
- **NFR20:** Application-логи (stdout/stderr)

#### Тестируемость (NFR21–NFR28)

- **NFR21:** 100% покрытие автотестами
- **NFR22:** Unit-тесты бэкенда
- **NFR23:** Интеграционные тесты API/БД/интеграции
- **NFR24:** UI-автотесты для всех интерфейсов
- **NFR25:** E2E: полное покрытие всех сценариев
- **NFR26:** Обязательно: тенант-изоляция, аудит-лог, RBAC
- **NFR27:** Идемпотентные тесты, работают на непустой БД
- **NFR28:** CI < 10 минут

#### Локализация (NFR29–NFR30)

- **NFR29:** Русский язык — основной
- **NFR30:** Архитектура поддерживает i18n с первого дня

#### Платформы (NFR31–NFR33)

- **NFR31:** Креаторы — Telegram Mini App
- **NFR32:** Бренды — веб-кабинет, десктоп
- **NFR33:** Админы — веб-панель, десктоп

### Дополнительные требования (из других секций PRD)

#### Юридические ограничения
- Трёхуровневая договорная схема (рамочный → заявка/Order → акт)
- Ст. 32 — нельзя передать права на все будущие произведения
- Маркировка рекламы (Приказ МИОР №59-НП)
- 4 раздельных согласия на обработку ПД (Закон N 94-V)
- Только 18+ (проверка по ИИН)

#### Бизнес-ограничения
- MVP — только бартер, без денежных расчётов
- Бренды создаются вручную (MVP)
- ~200 креаторов для EFW
- 1 разработчик, дедлайн ~20 апреля

#### Открытые вопросы (блокирующие техдизайн)
- OQ-3: метод верификации владения аккаунтом
- OQ-15: шифрование at rest по Закону N 94-V

## Оценка полноты PRD

### Сильные стороны

- **59 FR** охватывают все capability areas: от регистрации до аудита
- **33 NFR** с конкретными метриками (< 3 сек, < 10 мин CI, 200 пользователей)
- **5 User Journeys** покрывают happy path + edge cases + dispute
- Юридическая инфраструктура детально проработана (законы, статьи, схемы)
- Фазированный роадмап (MVP → Growth → Vision) с чётким must-have

### Пробелы и рекомендации

| # | Пробел | Критичность | Рекомендация |
|---|---|---|---|
| G1 | **OQ-3 не решён** — метод верификации владения аккаунтом не определён. FR5 ссылается на OQ-3. Без решения архитектор не может спроектировать этот flow | Высокая | Решить до начала техдизайна |
| G2 | **OQ-15 не решён** — шифрование at rest. NFR2 зависит от исследования | Средняя | Можно начать без решения (проектировать с возможностью включить) |
| G3 | **Нет данных о формах/полях** — FR26 говорит «бриф с ТЗ», но конкретные поля формы не описаны. UX-дизайнеру нужно будет их определить | Средняя | Покроется на этапе UX Design |
| G4 | **Нет описания статусной модели** — упоминаются статусы (на модерации, одобрен, отклонён, заблокирован; черновик, опубликована, завершена), но единая модель переходов не зафиксирована | Средняя | Покроется на этапе Architecture |
| G5 | **Нет API-контракта** — FR описывают capabilities, но нет mapping на эндпоинты | Ожидаемо | Покроется на этапе Architecture |
| G6 | **Email-провайдер не выбран** — FR51 требует email, NFR11 задаёт SLA, но интеграция не описана | Низкая | Решение на этапе техдизайна |
| G7 | **Нет описания data model** — сущности упоминаются (Creator, Brand, Campaign, Order, Content), но ERD/схема отсутствует | Ожидаемо | Покроется на этапе Architecture |

### Verdict

**PRD готов для перехода к следующим этапам** (Architecture, UX Design). Критических блокеров нет — OQ-3 (верификация аккаунта) можно решить параллельно с техдизайном. Все пробелы G3–G7 ожидаемы и будут закрыты на соответствующих этапах.
