# Тестирование [REQUIRED]

Тесты — это документация, которая проверяет сама себя. Качество тестов определяет уверенность в коде.

---

## CS-32: Каждый E2E-тест файл — описание в начале

**Scope:** both (Go E2E + Playwright E2E)

**Почему:** Без описания непонятно, какие бизнес-сценарии покрыты в файле. Новый разработчик (или AI-агент) не знает, нужно ли добавлять тест сюда или в другой файл.

**Плохо** (e2etest/audit_test.go:14):
```go
// --- Audit Log E2E ---
```

**Хорошо:**
```go
/*
Тесты аудит-лога.

Покрытие:
- Все операции (логин, создание/изменение/удаление бренда, назначение/удаление менеджера)
  создают записи аудита с корректными actor_id, action, entity_type, entity_id, old/new values
- Фильтрация по actor_id, entity_type, action, date range
- Пагинация (page, per_page, граничные значения)
- Доступ: только admin видит аудит-лог, brand_manager получает 403
- Невалидные фильтры возвращают 422
*/
package e2etest
```

**Для Playwright E2E:**
```typescript
/**
 * Тесты потока аутентификации бренд-менеджера.
 *
 * Покрытие:
 * - Логин с валидными credentials → редирект на дашборд
 * - Логин с невалидным паролем → ошибка
 * - Refresh token → новый access token без перелогина
 * - Logout → редирект на логин, повторный доступ к дашборду заблокирован
 */
```

**Правило:** каждый тест-файл начинается с многострочного комментария (`/* */` в Go, `/** */` в TypeScript), описывающего:
1. Что тестируется (какой бизнес-домен)
2. Какие сценарии покрыты (список)
3. Описание **обязано** соответствовать действительности — при изменении тестов обновлять

## CS-33: В тестах репозиториев — строковые литералы, не константы

**Scope:** backend

**Почему:** Если константа `ColActorID = "actorid"` (опечатка), а тест использует ту же константу — тест зелёный, но SQL падает в runtime. Строковые литералы в тестах — двойная проверка правильности констант.

**Плохо:**
```go
// repository/audit_test.go — те же константы что в коде
func TestAuditRepo_Create(t *testing.T) {
    mock.ExpectExec(regexp.QuoteMeta(
        `INSERT INTO ` + TableAuditLogs + ` (` + ColAuditActorID + `, ...)`)).
        // если константа неправильная, тест не поймает
```

**Хорошо:**
```go
func TestAuditRepo_Create(t *testing.T) {
    mock.ExpectExec(regexp.QuoteMeta(
        `INSERT INTO audit_logs (actor_id, actor_role, action, entity_type, entity_id, old_value, new_value, ip_address)`)).
        WithArgs(/* ... */).
        WillReturnResult(pgxmock.NewResult("INSERT", 1))
    // Строковые литералы здесь — намеренно!
    // Если константа ColAuditActorID != "actor_id", этот тест упадёт
```

**Правило:** тесты репозиториев проверяют точные SQL-строки с хардкоженными именами таблиц и колонок. Использование констант из production-кода в тестах запрещено — это обесценивает проверку.

## CS-34: Race detector — всегда включён

**Scope:** backend

**Почему:** Data race = undefined behavior. Может проявиться как случайный crash раз в 1000 запросов только в проде под нагрузкой. Race detector ловит это детерминированно.

**Текущее состояние:** уже включён в Makefile и CI (`-race` flag). Это правильно.

**Правило:** 
- `go test -race` во всех make-таргетах (unit, E2E backend)
- `-race` в CI pipeline
- **Отключение `-race` запрещено**, даже если тесты стали медленнее
- Если тест не проходит с `-race` — это баг в коде, не в тесте
