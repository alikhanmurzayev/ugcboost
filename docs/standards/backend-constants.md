# Константы вместо строковых литералов

Строковые литералы для enum-подобных значений — **критическая ошибка**. Опечатка в строке — silent bug: компилятор не поймает, тесты могут не покрыть, баг всплывёт в проде.

## Принцип

Любое значение, которое используется больше одного раза или имеет конечный набор вариантов — **только константа**. Определяется в одном месте, используется везде через импорт.

Примеры (список не исчерпывающий):
- Роли пользователей и другие enum-значения из OpenAPI-контракта
- Коды ошибок
- Имена HTTP-заголовков, cookie
- Названия таблиц и колонок в SQL-запросах
- Значения конфигурации с конечным набором вариантов (log level, environment и т.п.)

Если сгенерированные константы уже есть (из OpenAPI, из схемы БД и т.п.) — использовать их, не дублировать вручную.

## Исключение: тесты

Backend E2E тесты (`e2etest/`) — изолированный модуль. Они **не импортируют** внутренние пакеты проекта. Используют собственную кодогенерацию или собственные константы.

Unit-тесты репозиториев: SQL-строки пишутся **литералами** — это двойная проверка, что константы в коде корректны.

## Что ревьюить

- [blocker] Строковый литерал enum-значения в коде (роли, коды ошибок, статусы) — должна быть константа или импорт из generated.
- [blocker] Литерал имени колонки/таблицы вне `*_test.go` (в repo-тестах — наоборот, обязательны).
- [major] HTTP-заголовок / cookie name как литерал.
- [minor] Конфигурационное значение из конечного набора (log level, environment) как литерал.
