# Библиотеки вместо велосипедов [REQUIRED]

Если для задачи есть проверенная, активно поддерживаемая библиотека — используем её. Самописные решения пишем только когда нет адекватной альтернативы, и это решение явно обосновано.

## Принцип

Каждая строка кастомного кода — строка, которую надо поддерживать, тестировать и дебажить. Библиотека с тысячами звёзд и живым мейнтейнером уже прошла через это за нас. Писать свой велосипед там, где есть готовое решение — это долг, который копится молча и всплывает в самый неподходящий момент.

Правило применяется к любой устоявшейся области: инфраструктурный код (конфигурация, lifecycle, логирование, HTTP/SQL-утилиты), тестовые утилиты (моки БД, fixture-библиотеки, snapshot-инструменты), общие бизнес-хелперы (парсинг форматов, работа со временем и т.п.).

## Правило

Перед написанием любого утилитарного кода — проверить, есть ли готовое решение (Awesome Go, pkg.go.dev, npm). Если есть подходящая библиотека с активной поддержкой — использовать её. Если нет — написать своё, но обосновать почему (в комментарии или PR-описании): нет аналога, аналоги заброшены, аналог тянет неприемлемый транзитивный граф, и т.п.

Кастомные хелперы для стандартных задач (парсинг env vars, graceful shutdown, перехват SQL в тестах и т.п.) — не писать. Ошибки парсинга/валидации — не игнорировать молча: невалидный конфиг = приложение не стартует.

## Бизнес-логика

Бизнес-логика (хендлеры, сервисы, доменные правила) — пишется самостоятельно. Здесь специфика слишком прикладная, чтобы покрывалось библиотекой. Границу между «бизнес» и «инфра/утилиты» проводим по наличию и зрелости готового решения: если существует устоявшаяся библиотека для задачи — задача уходит в категорию «использовать библиотеку».
