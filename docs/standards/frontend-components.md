# Фронтенд: компоненты и UI

## Структура файлов

Feature-based: код организован по доменам, не по типам файлов.

```
src/
├── app/                  # App.tsx, providers, routing
├── features/             # по доменам
│   ├── auth/
│   │   ├── components/
│   │   └── hooks/
│   ├── brands/
│   │   ├── components/
│   │   └── hooks/
│   └── campaigns/
└── shared/               # переиспользуемое между фичами
    ├── components/       # Button, Skeleton, ErrorState
    ├── hooks/            # useDebounce, useLocalStorage
    ├── constants/        # routes, queryKeys
    └── utils/            # formatDate и т.п.
```

Каждая фича содержит свои компоненты, хуки и логику рядом. Общее — в `shared/`.

## i18n

Хардкод пользовательского текста в JSX запрещён. Все строки — через `react-i18next`.

- Переводы в JSON по неймспейсам, совпадающим с фичами: `locales/ru/auth.json`, `locales/ru/brands.json`
- Вложенные ключи: `t("auth.login.button")`, `t("brands.empty")`
- Простой текст — `t("key")`, текст с JSX внутри — компонент `Trans`
- Язык по умолчанию — русский. Архитектура готова к добавлению других языков

## Route paths — константы

Все route paths определяются в одном файле. Строковые литералы путей в компонентах и роутере запрещены. Для динамических путей — фабричные функции. Навигация содержит только существующие роуты — мёртвые ссылки запрещены.

## Роли — константы

Строковые литералы ролей в компонентах запрещены. Роли сравниваются через константы или типы из кодогенерации (см. стандарт типов).

## Декомпозиция компонентов

Компонент > 150 строк — сигнал к декомпозиции. Страница (Page) — оркестратор, который собирает подкомпоненты. Бизнес-логика (мутации, обработчики) живёт в подкомпонентах рядом с UI, который их использует.

## Нативные диалоги запрещены

`window.confirm()`, `window.alert()`, `window.prompt()` запрещены. Все диалоги — через компоненты UI-библиотеки. Нативные диалоги нельзя стилизовать, локализовать, они блокируют поток.

## Loading / Error / Empty states — обязательны

Каждый запрос данных обязан обрабатывать три состояния:
- **Loading** — skeleton или spinner, не голый текст
- **Error** — сообщение об ошибке с возможностью retry
- **Empty** — понятное сообщение что данных нет

Использовать переиспользуемые компоненты из shared.

## data-testid

Каждый интерактивный элемент и ключевой контейнер получает атрибут `data-testid` — контракт между компонентами и browser E2E тестами (см. стандарт browser E2E).

## Что ревьюить

- [major] Hardcoded JSX-строка для пользовательского текста (вместо `t("...")`).
- [major] `window.confirm` / `window.alert` / `window.prompt`.
- [major] Литерал route path в компоненте или роутере (вместо константы).
- [major] Литерал роли в компоненте (вместо константы или generated type).
- [minor] Loading / Error / Empty state не обработан для запроса данных.
- [minor] `data-testid` отсутствует на интерактивном элементе / ключевом контейнере.
- [minor] Компонент > 150 строк без декомпозиции.
