# Browser E2E тесты

Playwright. Только критические user flows — не дублируем edge cases из backend E2E.

## Структура файлов

```
frontend/
├── web/
├── tma/
├── landing/
└── e2e/
    ├── helpers/
    │   ├── api.ts         # seed, login, cleanup — HTTP к бэкенду
    │   ├── ui.ts          # общие UI-действия
    │   ├── ui-web.ts      # UI-действия специфичные для web
    │   └── ui-tma.ts      # UI-действия специфичные для tma
    ├── web/               # тесты для web-приложения
    ├── tma/               # тесты для TMA
    └── cross/             # cross-app flow (web + tma)
```

Тест-файлы внутри `e2e/web/`, `e2e/tma/`, `e2e/cross/` именуются по flow: например, `auth.spec.ts`, `campaign-lifecycle.spec.ts`.

## Нейминг

По user flow:

```typescript
test.describe('Campaign creation flow', () => {
  test('manager creates campaign for brand', ...)
  test('admin approves campaign', ...)
})
```

Допустимо по фиче, когда flow неестественен (например, проверка защиты роутов).

## Создание тестовых данных

- Тестовое API (`/test/*`) — только когда нет UI flow (создание admin, получение raw токенов)
- Всё остальное — через UI. Если есть форма — используем форму, не API

Для сценариев с несколькими пользователями — отдельный browser context на каждого. Переключение мгновенное, без logout/login.

## Хелперы

Composable — могут делать HTTP-запросы к бэкенду, вызывать другие хелперы, комбинировать шаги:
- `helpers/api.ts` — seed, login через API, cleanup
- `helpers/ui.ts` — общие UI-действия (заполнить форму, дождаться навигации)
- `helpers/ui-web.ts`, `helpers/ui-tma.ts` — специфичные для приложения

## Cleanup

Defer-based cleanup stack с управлением через env var `E2E_CLEANUP` (по умолчанию `true`):
- `E2E_CLEANUP=true` — данные удаляются через API после каждого теста в обратном порядке (LIFO, уважает foreign keys)
- `E2E_CLEANUP=false` — данные остаются для анализа состояния при дебаге упавших тестов

Удаление — через бизнес-ручки приоритетно. Если бизнес-ручки удаления нет — тестовая ручка (`/test/*`).

## Изоляция

- Параллельное выполнение поддерживается (Playwright workers). Количество workers настраивается в `playwright.config.ts` в зависимости от ресурсов машины. Тесты обязаны быть готовы к параллельному выполнению (изоляция данных)
- Каждый тест создаёт полный набор данных с нуля
- Уникальность через generated emails/names — тесты идемпотентны
- Тест проходит и с чистой БД, и с накопленными данными от предыдущих прогонов

## Локаторы

`data-testid` — стабильный контракт между фронтендом и тестами. Не привязан к тексту, не ломается при смене копирайта или i18n.

Каждый интерактивный элемент и ключевой контейнер в компонентах получает `data-testid`.

```typescript
await page.getByTestId('login-button').click()
await page.getByTestId('email-input').fill('test@example.com')
```

## Assertions

- Навигация: `expect(page).toHaveURL(...)`
- Видимость: `expect(locator).toBeVisible()`
- Текст: `expect(locator).toHaveText(...)`
- Корректность данных: проверяем что на странице отображаются правильные данные (имя бренда, email пользователя, статус кампании и т.п.), а не только что элемент видим

## Комментарий в начале файла

Обязателен. JSDoc-блок (`/** ... */`) в начале файла с полным описанием проверяемого user flow: что и зачем проверяется, какие данные создаются в `beforeAll`, как они удаляются в `afterAll`.

**Язык — русский.** Это локальное исключение из общего правила `naming.md` ("комментарии на английском"), действующее только на заголовочный JSDoc e2e-spec файла. Обычные inline-комментарии внутри `test()` (например, `Reload page (simulates F5 — token lost from memory)`) остаются на английском по общему правилу.

**Формат — связный нарратив, не нумерованный список шагов.** Заголовок рассказывает историю цикла: что каждый `test()`/`test.describe` проверяет и зачем (какой UX-инвариант или auth-свойство закрывает). Технические детали (tedтовые эндпоинты, `data-testid`-ключи, `E2E_CLEANUP`) встраиваются в повествование, а не выкидываются.

Один связный абзац или 2–3 абзаца по смысловым группам (happy path и ошибки, session restore и logout, защита роутов) — в зависимости от объёма `test.describe`. Нумерованные `Steps: 1. ... 2. ...` — запрещены.

### Пример

Плохо (нумерованный список шагов):

```typescript
/**
 * Auth flow E2E tests — web application.
 *
 * Steps:
 * 1. Seed admin user via /test/seed-user
 * 2. Happy login → dashboard with sidebar
 * 3. Wrong password → error, stay on login
 * ...
 */
```

Хорошо (нарратив):

```typescript
/**
 * Auth flow E2E — веб-приложение.
 *
 * Тесты проходят полный цикл аутентификации одного администратора. В
 * beforeAll сеем пользователя через POST /test/seed-user, в afterAll удаляем
 * его (если E2E_CLEANUP !== "false").
 *
 * Happy path: правильный email и пароль ведут на дашборд, в сайдбаре виден
 * email, роль «Админ» и полный набор навигационных ссылок. Затем проверяем
 * неправильный пароль: остаёмся на /login с сообщением «Неверный email или
 * пароль» без утечки того, какой именно компонент ошибся. ...
 */
```

## Landing (Astro) specifics

Лендос не имеет своего seed-bootstrap (в отличие от web/tma c admin-аккаунтом). Тесты не должны полагаться на конкретные значения, которые могут меняться: справочники, контент CMS, конкретные admin-emails. Assert'ы — на инварианты (форма submit'ится; redirect на success-page произошёл; обязательное поле помечено required), не на конкретные строки из БД.

## Что ревьюить

- [major] Локатор по тексту (`getByText("Войти")`) вместо `data-testid` — ломается при смене копирайта / i18n.
- [major] `t.describe` по фиче, а не user flow (когда flow естественен).
- [major] Hardcoded зависимость от seed-значения — тест ожидает `admin@example.com` или `gaming` в категориях, поломается если seed/словарь изменился.
- [major] `E2E_CLEANUP=false` оставлен в коммите (только локальный override).
- [minor] Header JSDoc не на русском, не нарратив (нумерованный список шагов).
- [minor] Edge case дублируется из backend e2e (frontend e2e — только критические user flows).
