# Unit-тесты фронтенда

Vitest + React Testing Library. Каждый слой тестируется изолированно.

## Нейминг

Файл теста рядом с тестируемым файлом: `useAuth.ts` → `useAuth.test.ts`, `formatDate.ts` → `formatDate.test.ts`.

`describe('{Module}', () => { ... })` для группировки, `it('should ...')` для сценариев. Сценарии идут по ходу исполнения кода: сначала ошибки и edge cases, в конце — happy path.

## По слоям

### Утилиты и хелперы
- Чистые функции: форматирование, валидация, трансформация данных
- Вход → выход, без моков
- Покрывать edge cases: null, undefined, пустые строки, граничные значения

### API-слой (хуки запросов)
- Мокаем сгенерированный API-клиент напрямую (не HTTP-уровень). MSW не нужен для unit-тестов — это overhead, MSW уместен для интеграционных тестов
- Проверяем: правильный endpoint вызван, правильные параметры переданы, ответ корректно обработан
- Проверяем обработку ошибок: сетевая ошибка, 4xx, 5xx — хук возвращает правильное состояние (error state)

### Кастомные хуки (бизнес-логика)
- `renderHook()` из React Testing Library
- Моки зависимостей (API-хуки, контексты)
- Проверяем: возвращаемые значения, state transitions, side effects

### Компоненты
- `render()` из React Testing Library
- Взаимодействие — через `userEvent` (не `fireEvent`). `userEvent` ближе к реальному поведению: фокус, blur, keyboard events
- Проверяем: правильные данные отображаются, взаимодействие (клик, ввод) вызывает правильные действия
- Обязательно тестировать: loading state, error state, empty state, данные отображены корректно
- Моки: дочерние компоненты не мокаем (если не тяжёлые), API-хуки мокаем
- Локаторы: предпочитать `getByRole`, `getByLabelText` (проверяют accessibility). `getByTestId` — когда нет подходящей роли или лейбла
- **Не тестируем**: стили, layout, CSS-классы. Только поведение и данные

### i18n в тестах
- Компоненты используют `t("key")` — в тестах оборачивать компонент в `I18nextProvider` с реальными переводами (не мокать). Тест проверяет что пользователь видит правильный текст, а не что вызван `t()` с правильным ключом

### Auth в тестах
- Компоненты зависящие от auth state — мокать Zustand store перед рендером. Задать нужного пользователя (роль, id) через мок, не через login flow
- RoleGuard и подобные — тестировать отдельно с разными ролями

## Async

Асинхронные операции (загрузка данных, мутации) — через `waitFor` и `findBy*` из React Testing Library. Не использовать `setTimeout` и ручные задержки.

## Изоляция

- Каждый тест изолирован — моки, состояние, DOM очищаются между тестами
- Данные создаются inline в каждом тесте

## Assertions

- `expect()` из Vitest
- Строгая проверка: не просто "компонент отрендерился", а конкретные данные на экране совпадают с ожидаемыми
- Проверка ошибок: error state отрендерился с правильным сообщением, toast/alert появился с правильным текстом. Проверять и факт появления, и содержание
- Snapshot testing запрещён — снапшоты хрупкие, обновляются бездумно, не проверяют поведение

## Coverage

Целевой порог — 80%. Исключения: index.ts (re-exports), конфигурационные файлы, сгенерированный код.

## Что ревьюить

- [blocker] Snapshot testing.
- [major] `fireEvent` вместо `userEvent` (последний ближе к реальному поведению — focus, blur, keyboard).
- [major] i18n замокан вместо реального `I18nextProvider` с переводами.
- [major] MSW для unit-теста (overhead — мокать API-клиент напрямую).
- [major] Loading / Error / Empty state не тестируется для компонента с запросом данных.
- [minor] Style / layout / CSS-class в ассерте (тестируем поведение и данные, не вёрстку).
