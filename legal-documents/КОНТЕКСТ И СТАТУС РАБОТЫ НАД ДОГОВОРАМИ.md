## 3. Юридическая модель (Агентская модель №2)

Это основа всей документации. Запомни логику — без неё ни один договор не читается.

**Кратко:** UGC boost — **агент**, действующий **от своего имени, но за счёт Бренда** (агентская модель по ст. 866 ГК РК / договор комиссии).

**Стороны и связи:**

```
       Бренд (Заказчик)                      Креатор (физлицо, ГПХ)
            │                                       │
            │  Агентский договор                    │  Рамочный договор + ТЗ
            │  (юрлицо ↔ юрлицо)                    │  (юрлицо ↔ физлицо)
            │                                       │
            └────────► UGC boost (ТОО Models Production) ◄────────┘
                       (агент / комиссионер / налоговый агент)
```

**Ключевые следствия:**
- Креатор и Бренд **не имеют прямого договора** между собой. Все обязательства идут через UGC boost.
- UGC boost — **налоговый агент** при выплатах креаторам-физлицам (удерживает ИПН, ОПВ, ВОСМС, СО). ОПВР (3.5% в 2026) — **наша внутренняя затрата**, не вычет из выплаты креатору.
- При спорах Креатор предъявляет претензии нам, мы — Бренду (через регресс). Бренд предъявляет претензии нам, мы — Креатору (через регресс).
- Известный системный риск — **неплатёжеспособность Креатора-физлица** (если Бренд через нас взыщет крупную компенсацию, Креатор может оказаться неплатёжеспособен). Митигация — через **4 защитные клаузулы в Агентском договоре с Брендом** (см. ПРИНЯТЫЕ РЕШЕНИЯ.md, решение #20).

**Почему именно эта модель** (а не №1 — где UGC boost просто витрина и креатор подписывает с Брендом напрямую):
- Контроль качества контента и кампаний (мы видим всё, можем модерировать).
- Единый канал для бренда (один договор, один акт, одна оплата).
- Единая точка налогового агентирования (бренду не нужно разбираться с физлицами в РК).
- Защита креаторов: они не подписывают сложные B2B-договоры с каждым брендом.

## 4. Архитектура документов

Три уровня:

### Уровень 1 — Публичные документы Платформы (общие для всех Пользователей)
- **Пользовательское соглашение (ПС)** — `ПОЛЬЗОВАТЕЛЬСКОЕ СОГЛАШЕНИЕ UGCBOOST.md`. Регулирует использование Платформы любым Пользователем (Креатор/Бренд/гость). Принимается через **click-wrap** при регистрации.
- **Политика обработки персональных данных** — `ПОЛИТИКА ОБРАБОТКИ ПЕРСОНАЛЬНЫХ ДАННЫХ UGCBOOST.md`. Принимается через **click-wrap**.
- Обновления через click-wrap акцепт новой редакции (см. решение #15 и #19).

### Уровень 2 — Рамочные договоры (по типам контрагентов)
- **Рамочный договор с Креатором** — `РАМОЧНЫЙ ДОГОВОР С КРЕАТОРОМ UGC BOOST.md`. Подписывается с каждым Креатором **через ЭЦП (TrustMe)** один раз. Бессрочный. Регулирует все будущие сделки.
- **Агентский договор с Брендом** — **ЕЩЁ НЕ СОЗДАН** (следующий артефакт работы). Подписывается с каждым Брендом через ЭЦП (TrustMe) один раз. Бессрочный.

### Уровень 3 — Технические задания (ТЗ) на конкретные сделки
- **ТЗ Креатору** — формируется в Платформе для каждой сделки. Креатор акцептует **через click-wrap** в интерфейсе Платформы (ст. 152, 395, 396 ГК РК). ЭЦП повторно не нужна.
- **ТЗ от Бренда** — аналогично, click-wrap в Платформе.

**Логика:** один ЭЦП-договор на старте, дальше всё работает через click-wrap внутри Платформы. Это резко снижает трение и масштабируется на сотни сделок в месяц.

**Логирование акцепта** (Алихан реализует на бэке): IP, время, версия документа, User-Agent — для каждого click-wrap. Нужно для NDA-санкций и любых юр. споров.

- Сторона — юрлицо, не физлицо. Нет налогового агента.

  1. Лимит ответственности UGC boost (наша комиссия + до X ₸ сверху).
  2. Бренд декларирует, что несёт ответственность за финальный креатив и кампанию.
  3. Прямой иск Бренда к Креатору — допустим (даём такую опцию).
  4. Обязательное страхование рисков (для крупных кампаний).
- **Решение #21** — ответственность за содержание рекламной кампании на стороне Бренда.
- **Решение #22** — ОПВР как наша внутренняя затрата, не Бренда.

### 5.5. ТЗ-шаблон (форма Технического задания) — ❌ НЕ НАЧАТ

Должно содержать поля:
- Идентификатор ТЗ, Бренд, Креатор.
- Описание задачи (форматы: Reels/TikTok/Stories/YouTube Shorts/Threads/фото/обзоры/тексты).
- Требования к контенту (длительность, упоминания, hashtags, mentions, SCM, метки рекламы).
- Сроки (создание, согласование сценария, публикация, неудаление).
- Вознаграждение (денежное или бартерное; стоимость указывается).
- Условия бартера (если применимо): кто отправляет, когда, кто оплачивает обратную доставку при отмене.

В Рамочном договоре уже есть отсылки к ТЗ, поэтому ТЗ должно соответствовать терминологии Рамочного.

## 8. Технические детали (генерация PDF)

PDF генерируется через **pandoc → HTML → Chrome headless**.

**CSS:** `legal-documents/_pdf/style.css` (закоммичено в этом коммите). Шрифты: Georgia (body, serif), Helvetica (заголовки), красный акцент `#c8102e`. Формат A4, поля 2cm.

**Команда генерации (выполняется из репо):**

```bash
# 1) MD → HTML с CSS
pandoc "legal-documents/РАМОЧНЫЙ ДОГОВОР С КРЕАТОРОМ UGC BOOST.md" \
  --standalone \
  --metadata title="РАМОЧНЫЙ ДОГОВОР UGC BOOST" \
  --css "legal-documents/_pdf/style.css" \
  -o "/tmp/contract.html"

# 2) HTML → PDF через Chrome headless
google-chrome --headless --disable-gpu \
  --print-to-pdf="/tmp/contract.pdf" \
  --no-pdf-header-footer \
  "file:///tmp/contract.html"

# 3) Положить итог рядом с MD
cp "/tmp/contract.pdf" "legal-documents/РАМОЧНЫЙ ДОГОВОР С КРЕАТОРОМ UGC BOOST.pdf"
```

**Известная косметическая проблема:** в шапке PDF отображаются **два заголовка подряд** — pandoc-метатайтл `РАМОЧНЫЙ ДОГОВОР UGC BOOST` (наверху) и собственный H1 договора `РАМОЧНЫЙ ДОГОВОР ОБ ОКАЗАНИИ УСЛУГ ПО СОЗДАНИЮ UGC-КОНТЕНТА`. Аидана пока не решала — оставлять или скрывать. Решения:
- (а) Убрать `--metadata title=...` — pandoc вставит первый H1 как title (но title всё равно появится).
- (б) Добавить в `style.css` `header.title-block-header { display: none; }` — спрячет блок с титлом метаданных.
- (в) Оставить как есть — две строки в шапке.
